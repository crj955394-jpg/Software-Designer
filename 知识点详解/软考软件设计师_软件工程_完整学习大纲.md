## 第一章 软件工程基础（★★ 必考）

### 1.1 软件危机与软件工程的提出

#### 1.1.1 软件危机（Software Crisis）

**教材定义**：
> 软件危机是指在计算机软件的开发和维护过程中所遇到的一系列严重问题。这些问题绝不仅仅是"不能正常运行的软件"，还包括如何开发软件、如何维护数量不断膨胀的已有软件等。

**软件危机的典型表现**（教材列举）：
1. 软件开发成本和进度的估计常常很不准确
2. 用户对"已完成的"软件系统不满意的现象经常发生
3. 软件产品的质量往往靠不住
4. 软件常常是不可维护的
5. 软件通常没有适当的文档资料
6. 软件成本在计算机系统总成本中所占的比例逐年上升
7. 软件开发生产率提高的速度远远跟不上计算机应用迅速普及深入的趋势

**产生原因**：
- 软件本身的特点：逻辑复杂、不可见、维护困难
- 软件开发方法不正确：缺乏正确的理论指导、不重视需求分析和评审

#### 1.1.2 软件工程（Software Engineering）

**教材定义**：
> 软件工程是指导计算机软件开发和维护的工程学科。采用工程的概念、原理、技术和方法来开发与维护软件，把经过时间考验而证明正确的管理技术和当前能够得到的最好的技术方法结合起来，以经济地开发出高质量的软件并有效地维护它。

**软件工程三要素**（★ 常考）：

| 要素 | 含义 |
|------|------|
| **方法** | 完成软件开发各项任务的技术方法（结构化方法、面向对象方法等） |
| **工具** | 为方法的运用提供自动或半自动的软件支撑环境 |
| **过程** | 获得高质量软件所需要完成的一系列任务的框架 |

### 1.2 软件生命周期（Software Life Cycle）

**教材定义**：
> 软件生命周期是指软件产品从提出、实现、使用维护到停止使用退役的整个过程。一般包括可行性研究与计划、需求分析、软件设计、编码、测试、维护等活动。

**软件生命周期各阶段**（经典三时期划分）：

| 时期 | 阶段 | 主要任务 |
|------|------|----------|
| **软件定义时期** | 可行性研究与计划 | 确定总目标、可行性、资源、进度 |
| | 需求分析 | 确定系统"做什么"，编写需求规格说明书 |
| **软件开发时期** | 概要设计 | 确定系统"怎么做"，设计软件结构（模块划分） |
| | 详细设计 | 设计每个模块的算法和数据结构 |
| | 编码 | 将设计翻译为程序代码 |
| | 测试 | 发现并纠正错误 |
| **软件运行维护时期** | 运行与维护 | 改正性、适应性、完善性、预防性维护 |

> ⚠️ **注意**：需求分析阶段确定"做什么"（What），概要设计阶段确定"怎么做"（How）。这是考试常考辨析点。

### 1.3 软件过程模型（★★★ 超高频考点）

**概念**：
> 软件过程模型（也称软件开发模型、软件生存周期模型）是软件开发全部过程、活动和任务的结构框架。它能直观地表达软件开发的全过程，明确规定要完成的主要活动和任务。

#### ① 瀑布模型（Waterfall Model）

**教材描述**：
> 瀑布模型将软件生存周期的各项活动规定为依线性顺序连接的若干阶段。它规定了由前至后、相互衔接的固定次序，如同瀑布流水逐级下落。

```
需求分析 → 概要设计 → 详细设计 → 编码 → 测试 → 运行维护
```

| 特点 | 说明 |
|------|------|
| **核心思想** | 线性顺序、阶段评审、文档驱动 |
| **优点** | 过程清晰、易于管理、强调文档、每个阶段有评审 |
| **缺点** | 不适应需求变化（最大缺点）、用户直到后期才能看到产品 |
| **适用场景** | 需求明确、技术成熟、规模较小的项目 |

> 🎯 **考试关键词**：线性顺序、文档驱动、不适应需求变化、每个阶段有评审

#### ② 增量模型（Incremental Model）

**教材描述**：
> 增量模型融合了瀑布模型的基本成分和原型实现的迭代特征。它将软件看作一系列增量构件，每个增量均发布一个可操作的产品。

```
增量1（核心功能）→ 增量2（附加功能）→ 增量3（更多功能）→ ... → 完整产品
```

| 特点 | 说明 |
|------|------|
| **核心思想** | 分批交付，每个增量是一个可运行的子系统 |
| **优点** | 较短时间内提交可完成部分工作的产品；用户有反馈机会；降低风险 |
| **缺点** | 需要开放的体系结构；增量构件划分困难 |
| **关键点** | 每个增量必须发布一个可操作的产品 |

> ⚠️ **与瀑布的区别**：瀑布是全部完成后交付；增量是分批次交付。

#### ③ 演化模型 — 原型模型（Prototype Model）

**教材描述**：
> 原型模型的第一步是建造一个快速原型，实现用户与系统的交互，用户对原型进行评价，进一步细化待开发软件的需求。通过逐步调整原型使其满足用户的要求，开发人员可以确定用户的真正需求是什么。

```
快速分析 → 构造原型 → 运行/评价原型 → 修改 → ... → 调整 → 最终产品
```

| 特点 | 说明 |
|------|------|
| **核心思想** | 快速构建原型，获取用户反馈，明确需求 |
| **优点** | 克服瀑布模型的缺点，减少需求不明确带来的风险 |
| **缺点** | 原型可能被当作最终产品使用（质量隐患） |
| **适用场景** | **需求不明确**的项目（★ 高频考点） |

> 🎯 **原型的分类**（教材）：
> - **抛弃型原型**：探索可行性，用完即弃
> - **演化型原型**：逐步演进为最终产品

#### ④ 螺旋模型（Spiral Model）

**教材描述**：
> 螺旋模型将瀑布模型和快速原型模型结合起来，强调了其他模型所忽视的风险分析。它沿着螺旋线旋转，每次旋转分为四个象限：制定计划、风险分析、实施工程、客户评估。

```
        风险分析
       ↗        ↘
  制定计划  →  实施工程
       ↖        ↙
        客户评估
```

| 特点 | 说明 |
|------|------|
| **核心思想** | 风险驱动 + 原型 + 瀑布的结合 |
| **四个象限** | ①制定计划 ②风险分析 ③实施工程 ④客户评估 |
| **优点** | 强调风险分析，适合大型复杂项目 |
| **缺点** | 需要丰富的风险评估经验，成本高 |
| **适用场景** | 大规模、高风险、需求不明确的项目 |

> 🎯 **螺旋模型最大特点**：引入了**风险分析**（考试最爱考的点）

#### ⑤ V 模型（V-Model）

**教材描述**：
> V 模型是瀑布模型的变种，它强调测试过程与开发过程的对应关系。左边是开发过程各阶段，右边是测试过程各阶段。

```
需求分析  ←——————————→  验收测试
概要设计  ←——————————→  系统测试
详细设计  ←——————————→  集成测试
  编码    ←——————————→  单元测试
```

| 特点 | 说明 |
|------|------|
| **核心思想** | 测试与开发各阶段一一对应 |
| **优点** | 强调测试的重要性，测试贯穿开发全过程 |
| **缺点** | 仍是线性模型，不适应需求变化 |
| **对应关系** | 需求分析↔验收测试、概要设计↔系统测试、详细设计↔集成测试、编码↔单元测试 |

#### ⑥ 喷泉模型（Fountain Model）

**教材描述**：
> 喷泉模型是一种以用户需求为动力，以对象为驱动的模型，主要用于描述面向对象的开发过程。

| 特点 | 说明 |
|------|------|
| **核心思想** | 面向对象、迭代、无间隙 |
| **优点** | 各阶段相互迭代、无缝衔接，提高开发效率 |
| **缺点** | 需要大量开发人员，文档管理困难 |
| **关键词** | 面向对象、迭代、无间隙活动 |

> 🎯 **喷泉模型最大特点**：面向对象开发模型，各阶段**迭代**且**无间隙**（不像瀑布有明显分界线）

#### ⑦ 基于构件的开发模型（CBD/CBSD）

**教材描述**：
> 基于构件的开发是利用预先包装的软件构件来构造应用系统。开发活动从需求分析开始，经过构件的合格性检验/调整/修改后，再进行构件的开发/修改。

| 特点 | 说明 |
|------|------|
| **核心思想** | 复用已有构件组装系统 |
| **优点** | 软件复用性高、提高开发效率和质量 |
| **缺点** | 构件库的质量和丰富程度制约开发 |

#### ⑧ 快速应用开发模型（RAD）

**教材描述**：
> RAD（Rapid Application Development）是瀑布模型的高速变种，通过大量使用可复用构件，采用基于构件的建造方法进行快速开发。

| 特点 | 说明 |
|------|------|
| **核心思想** | 构件复用 + 代码生成器 + 短周期 |
| **适用** | 适于管理信息系统的开发 |

#### ⑨ 统一过程模型（RUP / UP）

**教材描述**：
> 统一过程（Rational Unified Process, RUP）是一种以用例驱动、以体系结构为核心、迭代及增量的软件开发过程。

**RUP 的核心特点**（★★ 常考）：

| 特点 | 说明 |
|------|------|
| **用例驱动** | 以用户需求的用例为中心驱动整个开发过程 |
| **以体系结构为核心** | 强调早期建立稳定的体系结构 |
| **迭代和增量** | 每次迭代产生一个增量版本 |

**RUP 的四个阶段**：

| 阶段 | 主要活动 |
|------|----------|
| **初始阶段** | 建立业务模型、定义项目范围 |
| **细化阶段** | 分析问题域、建立体系结构框架 |
| **构建阶段** | 开发构件和应用程序、集成为产品 |
| **移交阶段** | 测试、制作发布版本、用户培训 |

#### ⑩ 敏捷开发方法（★★★ 近年高频考点）

**教材定义**：
> 敏捷开发是一种以人为核心、迭代、循序渐进的开发方法。在敏捷开发中，软件项目的构建被切分成多个子项目，各个子项目成果都经过测试，具备集成和可运行的特征。

**敏捷宣言**（四个价值观）：
1. **个体和互动** 高于 流程和工具
2. **工作的软件** 高于 详尽的文档
3. **客户合作** 高于 合同谈判
4. **响应变化** 高于 遵循计划

**常见敏捷方法**：

| 方法 | 特点 |
|------|------|
| **极限编程（XP）** | 结对编程、测试驱动开发、持续集成、小型发布、简单设计 |
| **Scrum** | Sprint 冲刺、每日站会、Product Backlog、Scrum Master |
| **水晶方法** | 不同项目规模使用不同"颜色"的方法族 |
| **特征驱动开发（FDD）** | 以特征为中心的迭代开发 |

**XP 的核心实践**（★ 常考）：

| 实践 | 含义 |
|------|------|
| **结对编程** | 两人共用一台电脑编程 |
| **测试驱动开发（TDD）** | 先写测试用例再写代码 |
| **持续集成** | 频繁集成代码，每次集成都构建和测试 |
| **小型发布** | 频繁发布小版本 |
| **简单设计** | 只满足当前需求的最简设计 |
| **重构** | 不改变外部行为的前提下改进内部结构 |
| **代码共有** | 集体代码所有权 |

> 🎯 **高频对比**：瀑布模型 vs 敏捷方法
> - 瀑布：计划驱动、文档驱动、线性、适合需求明确
> - 敏捷：人本驱动、代码驱动、迭代、适合需求不确定

#### ⑪ 过程模型对比总表（★★★ 必背）

| 模型 | 核心特点 | 优点 | 缺点 | 适用场景 |
|------|----------|------|------|----------|
| **瀑布** | 线性顺序 | 清晰可控 | 不适应变化 | 需求明确 |
| **增量** | 分批交付 | 早期交付 | 架构需开放 | 需求较明确 |
| **原型** | 快速原型 | 明确需求 | 原型质量差 | **需求不明确** |
| **螺旋** | 风险分析 | 降低风险 | 成本高 | 大型高风险 |
| **V 模型** | 测试对应 | 强调测试 | 仍是线性 | 对测试要求高 |
| **喷泉** | 面向对象/迭代 | 无缝衔接 | 管理困难 | 面向对象 |
| **RAD** | 构件复用 | 开发快 | 依赖构件 | 信息系统 |
| **RUP** | 用例/体系结构/迭代 | 规范可控 | 较重 | 中大型项目 |
| **敏捷** | 人本/迭代/响应变化 | 灵活高效 | 难以度量 | 需求多变 |

### 📝 本章典型真题

**【2023年真题改编】**
> 软件开发模型用于指导软件的开发。瀑布模型的突出特点是（）。
> A. 严格定义了各阶段提交的文档
> B. 严格定义了各阶段提交的产品
> C. 强调风险分析
> D. 不适应需求变化

**答案：D**（部分教材版本选 A，视出题角度而定）
**解析**：瀑布模型最大的问题就是不适应需求变化，这也是原型模型和敏捷方法被提出的动机。

**【2022年真题改编】**
> （）适用于需求不明确的软件开发。
> A. 瀑布模型　　B. 增量模型　　C. 原型模型　　D. 喷泉模型

**答案：C**
**解析**：原型模型的核心目的就是通过快速原型明确需求，专门针对需求不明确的场景。

**【2021年真题改编】**
> 在开发一个系统时，如果用户对系统的目标不是很清楚，难以定义需求，最好采用（）。
> A. 瀑布模型　　B. 原型模型　　C. V 模型　　D. 螺旋模型

**答案：B**（D 螺旋模型也可接受，但原型模型是最直接针对需求不明确的）
**解析**：需求不明确时首选原型模型，通过快速原型获取用户反馈明确需求。

**【2020年真题改编】**
> 螺旋模型综合了（）的优点，并增加了风险分析。
> A. 瀑布模型和演化模型　　B. 瀑布模型和喷泉模型
> C. 演化模型和喷泉模型　　D. 增量模型和喷泉模型

**答案：A**
**解析**：螺旋模型综合了瀑布模型（结构化、阶段性）和演化模型/原型模型（迭代）的优点，加上了风险分析。

## 第二章 需求工程（★★★ 重点）

### 2.1 需求工程概述

**教材定义**：
> 需求工程是指创建和维护系统需求文档所必须的一系列活动。需求工程包括需求获取、需求分析与协商、需求规格说明、需求验证和需求管理。

**需求工程的五个阶段**：

| 阶段 | 主要活动 |
|------|----------|
| **需求获取** | 通过访谈、问卷、观察、原型等方法获取用户需求 |
| **需求分析** | 分析需求的可行性、一致性和完备性，发现和解决冲突 |
| **需求规格说明** | 编写软件需求规格说明书（SRS），用规范化方式描述需求 |
| **需求验证** | 验证需求的正确性、一致性、完整性、可验证性 |
| **需求管理** | 在开发过程中管理需求的变更 |

### 2.2 需求的分类（★★★ 高频考点）

| 分类 | 说明 | 举例 |
|------|------|------|
| **功能需求** | 系统应该做什么（功能） | 系统应能处理用户登录 |
| **非功能需求** | 系统的运行特性（性能、可靠性等） | 响应时间 ≤ 2秒，可用性 99.9% |
| **设计约束** | 限制设计决策的条件 | 必须使用 Java 语言、必须兼容 Windows |

**非功能需求的细分**（教材）：

| 类型 | 含义 |
|------|------|
| **产品需求** | 产品的行为（性能、可靠性、安全性等） |
| **机构需求** | 用户所在机构的政策和规定（开发标准、交付要求等） |
| **外部需求** | 来自系统外部的因素（法律法规、道德规范、互操作性等） |

### 2.3 需求分析的方法

#### 2.3.1 结构化分析方法（SA）

**教材定义**：
> 结构化分析方法（Structured Analysis, SA）是一种面向数据流进行需求分析的方法。它使用数据流图（DFD）、数据字典（DD）、判定表和判定树等工具，以自顶向下、逐层分解的方式分析系统。

**结构化分析的三大工具**：

| 工具 | 用途 |
|------|------|
| **数据流图（DFD）** | 描述系统中数据的流动和处理过程 |
| **数据字典（DD）** | 定义数据流图中所有数据元素的含义 |
| **判定表/判定树** | 描述加工逻辑中的条件组合 |

#### 2.3.2 数据流图（DFD）（★★★ 下午题必考 15 分）

**DFD 的基本元素**：

| 元素 | 符号 | 含义 |
|------|------|------|
| **外部实体** | 方框 | 系统外部的数据源或终点（人、组织、外部系统） |
| **加工（处理）** | 圆圈/圆角矩形 | 对数据进行变换处理 |
| **数据存储** | 开口矩形/双横线 | 数据保存的地方（文件、数据库） |
| **数据流** | 带箭头的线 | 数据的流动方向 |

**DFD 的层次结构**（自顶向下逐层细化）：
```
顶层图（上下文图/0层图）
  └── 1层图（加工的分解）
       └── 2层图（进一步分解）
            └── ...
```

**绘制 DFD 的原则**（★★★ 下午题常考）：
1. **守恒原则**：加工的输入数据流必须能产生输出数据流（数据不能凭空产生或消失）
2. **父图与子图平衡**：子图的输入输出数据流必须与父图中对应加工的输入输出一致
3. **数据守恒**：一个加工的输出数据必须从输入数据或加工中产生
4. **外部实体不能直接与数据存储交互**（需通过加工）
5. **加工必须既有输入又有输出**

> 🎯 **下午题 DFD 解题技巧**：
> - 找缺失的数据流：对比父图子图，检查输入输出是否一致
> - 找加工的命名：动词+名词（如"处理订单"）
> - 找数据存储的命名：名词（如"订单文件"）
> - 检查是否有数据守恒违规

#### 2.3.3 数据字典（DD）

**教材定义**：
> 数据字典是对数据流图中包含的所有元素的定义的集合。它是 DFD 的补充工具，为 DFD 中出现的每个数据流、数据存储、加工、外部实体提供精确的定义。

**数据字典的组成**：
- 数据流条目：名称、组成、来源、去向
- 数据项条目：名称、类型、长度、取值范围
- 数据存储条目：名称、组成、关键字
- 加工条目：加工编号、名称、处理逻辑

#### 2.3.4 判定表与判定树（★★ 常考）

**判定表**：

| 条件 | 规则1 | 规则2 | 规则3 | 规则4 |
|------|-------|-------|-------|-------|
| 条件1 | Y | Y | N | N |
| 条件2 | Y | N | Y | N |
| **动作** | **A** | **B** | **C** | **D** |

**判定树**：用树形结构表示条件组合和对应动作的逻辑关系，更直观。

**两者对比**：

| 对比项 | 判定表 | 判定树 |
|--------|--------|--------|
| 直观性 | 一般 | 更直观 |
| 条件数多时 | 清晰（表格规整） | 树形庞大难读 |
| 适合场景 | 条件组合多、逻辑复杂 | 条件少、逻辑简单 |

## 第三章 系统设计（★★★ 重点）

### 3.1 软件设计的两个阶段

**教材划分**：

| 阶段 | 别称 | 任务 | 主要产出 |
|------|------|------|----------|
| **概要设计** | 总体设计、结构设计 | 确定系统由哪些模块组成、模块间的关系 | 模块结构图、接口说明 |
| **详细设计** | 过程设计 | 确定每个模块的算法、数据结构 | 程序流程图、伪代码 |

### 3.2 结构化设计（SD）概要设计（★★★ 核心）

#### 3.2.1 模块化设计原则

**模块独立性**是衡量软件设计好坏的关键标准，用两个定性指标度量：

| 指标 | 含义 | 理想状态 |
|------|------|----------|
| **耦合（Coupling）** | 模块之间相互依赖的程度 | 越低越好 |
| **内聚（Cohesion）** | 模块内部各元素之间联系的紧密程度 | 越高越好 |

**耦合类型**（从低到高，★★★ 必背）：

| 类型 | 说明 | 程度 |
|------|------|------|
| **非直接耦合** | 两个模块无直接关系 | 最低（最好） |
| **数据耦合** | 模块间通过参数传递数据 | 低 |
| **标记耦合** | 模块间通过参数传递数据结构 | 较低 |
| **控制耦合** | 模块间传递控制信息（如开关量） | 中等 |
| **公共耦合** | 模块间共享公共数据区 | 较高 |
| **内容耦合** | 一个模块直接访问另一模块的内部数据或代码 | 最高（最差） |

> 🎯 **考试技巧**：题目问"哪种耦合最好/最差"——
> - 最好：非直接耦合/数据耦合
> - 最差：内容耦合

**内聚类型**（从低到高，★★★ 必背）：

| 类型 | 说明 | 程度 |
|------|------|------|
| **偶然内聚** | 模块内各元素毫无关系，只是碰巧在一起 | 最低（最差） |
| **逻辑内聚** | 模块内元素逻辑功能相似但实际不同 | 低 |
| **时间内聚** | 模块内元素在同一时间段执行 | 较低 |
| **过程内聚** | 模块内元素按特定次序执行 | 中等 |
| **通信内聚** | 模块内元素操作同一数据集 | 较高 |
| **顺序内聚** | 模块内元素前一个的输出是后一个的输入 | 高 |
| **功能内聚** | 模块内所有元素共同完成一个单一功能 | 最高（最好） |

> 🎯 **考试技巧**：题目问"哪种内聚最好/最差"——
> - 最好：功能内聚
> - 最差：偶然内聚

#### 3.2.2 模块设计启发式规则（教材指导性原则）

1. 改进软件结构提高模块独立性
2. 模块规模适中（通常 50–100 行）
3. 深度、宽度、扇出、扇入应适当
   - **扇出**：一个模块直接调用的模块数（3–9 为宜）
   - **扇入**：有多少模块直接调用本模块
4. 模块的作用域应在控制域之内
5. 降低模块接口的复杂度
6. 设计单入口单出口的模块
7. 模块功能应该可预测

#### 3.2.3 系统结构图（SC）

**结构图的基本组成**：

| 元素 | 含义 |
|------|------|
| **模块** | 用矩形表示 |
| **调用关系** | 用箭头表示模块间的调用 |
| **数据传递** | 箭头旁标注传递的数据 |
| **控制信息** | 带空心圆标记的箭头 |

### 3.3 详细设计的工具（★★★ 高频考点）

| 工具 | 类型 | 特点 |
|------|------|------|
| **程序流程图** | 图形 | 最常用，直观，但不支持逐步求精 |
| **N-S 图（盒图）** | 图形 | 强制结构化，不允许随意跳转 |
| **PAD 图（问题分析图）** | 图形 | 支持自顶向下逐步求精，清晰 |
| **判定表** | 表格 | 适合多条件组合的逻辑 |
| **判定树** | 图形 | 直观，适合简单逻辑 |
| **PDL（伪代码）** | 文字 | 类似编程语言的描述方式 |
| **HIPO 图** | 图形 | 层次图 + 输入/处理/输出图 |

**详细设计工具对比**（★ 常考）：

| 对比项 | 程序流程图 | N-S 图 | PAD 图 |
|--------|-----------|--------|--------|
| 是否支持逐步求精 | 不支持 | 支持 | 支持 |
| 是否强制结构化 | 不强制 | 强制 | 支持 |
| 直观性 | 最好 | 一般 | 较好 |
| 是否容易转换为代码 | 一般 | 较易 | 较易 |

### 3.4 人机界面设计原则

**教材提出的三条黄金规则**（由 Theo Mandel 提出）：

1. **置用户于控制之下**：以不强迫用户进入不必要或不希望的动作的方式来定义交互模式
2. **减少用户的记忆负担**：界面应减少用户对记忆的要求
3. **保持界面一致**：系统内部和系统外部应保持一致

### 📝 本章典型真题

**【2023年真题改编】**
> 模块 A 直接访问模块 B 的内部数据，则这两个模块之间的耦合类型为（）。
> A. 数据耦合　　B. 标记耦合　　C. 公共耦合　　D. 内容耦合

**答案：D**
**解析**：直接访问另一个模块的内部数据属于内容耦合，是耦合程度最高的类型。

**【2022年真题改编】**

> 在详细设计阶段，不支持逐步求精的设计工具是（）。

**答案：A**
**解析**：程序流程图中的箭头可以随意跳转，不支持自顶向下的逐步求精。N-S 图和 PAD 图都支持。

## 第四章 面向对象方法与 UML（★★★ 核心重点）

### 4.1 面向对象基本概念

| 概念 | 定义 | 举例 |
|------|------|------|
| **对象** | 系统中用来描述客观事物的一个实体，是构成系统的基本单位 | 一个学生对象 |
| **类** | 一组具有相同属性和行为的对象的抽象 | Student 类 |
| **封装** | 将对象的属性和方法结合在一起，对外隐藏内部实现细节 | private 属性 + public 方法 |
| **继承** | 子类自动拥有父类的属性和方法 | GraduateStudent 继承 Student |
| **多态** | 同一操作作用于不同对象产生不同的执行结果 | 方法重写/重载 |
| **消息** | 对象之间通信的机制 | 对象 A 向对象 B 发送消息 |
| **聚合/组合** | 整体与部分的关系 | 汽车和发动机 |

**继承 vs 聚合 vs 组合**（★ 常考辨析）：

| 关系 | 特点 | 生命周期 |
|------|------|----------|
| **继承（泛化）** | is-a 关系（学生是人） | — |
| **聚合** | has-a 关系，部分可独立存在 | 部分可独立于整体 |
| **组合** | has-a 关系，部分不可独立存在 | 部分与整体同生命周期 |

### 4.2 UML 概述

**UML（Unified Modeling Language，统一建模语言）定义**：
> UML 是一种可视化建模语言，用于对软件系统进行描述、可视化处理、构造和建立软件系统制品的文档。

**UML 的图分为两大类**（★★★ 常考分类）：

| 类别 | 包含的图 |
|------|----------|
| **静态图（结构图）** | 类图、对象图、构件图、部署图、包图、复合结构图、制品图 |
| **动态图（行为图）** | 用例图、状态图、活动图、顺序图、通信图、定时图、交互概览图 |

> 🎯 **常考简化分类**（考试常用）：
> - 静态：类图、对象图、构件图、部署图
> - 动态：用例图、状态图、活动图、顺序图（序列图）、通信图（协作图）

### 4.3 UML 各类图详解（★★★ 核心）

#### 4.3.1 用例图（Use Case Diagram）（★★★ 下午题常考）

**作用**：描述系统的功能需求，展示参与者（Actor）与用例（Use Case）之间的关系。

**基本元素**：

| 元素 | 符号 | 含义 |
|------|------|------|
| **参与者（Actor）** | 小人图形 | 与系统交互的外部实体（用户、外部系统） |
| **用例（Use Case）** | 椭圆 | 系统提供的功能 |
| **系统边界** | 矩形框 | 划定系统范围 |
| **关系** | 连线/箭头 | 参与者与用例之间的关联 |

**用例间的关系**（★★★ 高频考点）：

| 关系 | 含义 | 箭头方向 | 示例 |
|------|------|----------|------|
| **包含（include）** | 基用例**必须**执行被包含用例 | 基用例 → 被包含用例 | "取款"包含"验证身份" |
| **扩展（extend）** | 在特定条件下**可选**执行扩展用例 | 扩展用例 → 基用例 | "打印凭条"扩展"取款" |
| **泛化（Generalization）** | 用例之间的继承关系 | 子用例 → 父用例 | "在线支付"泛化自"支付" |

> 🎯 **包含 vs 扩展的区分**（必考）：
> - 包含：无条件必须执行（箭头指向被包含者）
> - 扩展：有条件才执行（箭头指向被扩展者）

#### 4.3.2 类图（Class Diagram）（★★★ 下午题常考 15 分）

**作用**：描述系统的静态结构，展示类、接口及其关系。

**类的表示**：
```
┌──────────────────┐
│    ClassName      │  ← 类名
├──────────────────┤
│ - attr1: Type     │  ← 属性（- private, + public, # protected）
│ + attr2: Type     │
├──────────────────┤
│ + method1(): void │  ← 方法
│ - method2(p: int) │
└──────────────────┘
```

**类之间的关系**（★★★ 必背）：

| 关系 | UML 符号 | 含义 | 强度 |
|------|----------|------|------|
| **依赖** | 虚线箭头 - - -> | 一个类使用另一个类（临时性） | 最弱 |
| **关联** | 实线箭头 ——→ | 结构性关系，长期持有引用 | 弱 |
| **聚合** | 实线+空心菱形 ◇—— | 整体-部分，部分可独立 | 中 |
| **组合** | 实线+实心菱形 ◆—— | 整体-部分，部分不可独立 | 强 |
| **泛化（继承）** | 实线+空心三角形 △—— | is-a 关系 | 最强 |
| **实现** | 虚线+空心三角形 △- -> | 类实现接口 | 最强 |

> 🎯 **关系强弱排序**（必背）：
> 依赖 < 关联 < 聚合 < 组合 < 泛化/实现

#### 4.3.3 顺序图 / 序列图（Sequence Diagram）（★★）

**作用**：描述对象之间的交互，强调消息的时间顺序。

**基本元素**：

| 元素 | 含义 |
|------|------|
| **对象/生命线** | 竖直虚线表示对象在时间轴上的存在 |
| **消息** | 水平箭头表示对象间的消息传递 |
| **激活条** | 矩形条表示对象正在执行 |

**消息类型**：
- 同步消息：实心箭头 →（等待返回）
- 异步消息：开放箭头 →（不等待返回）
- 返回消息：虚线箭头 - ->

#### 4.3.4 通信图 / 协作图（Communication Diagram）

**作用**：描述对象之间的交互，强调对象之间的组织结构（与顺序图表达的信息等价，但视角不同）。

#### 4.3.5 状态图（State Diagram）（★★）

**作用**：描述对象在其生命周期内状态的变化。

**基本元素**：

| 元素 | 含义 |
|------|------|
| **初始状态** | 实心圆 ● |
| **终止状态** | 实心圆加外圈 ◎ |
| **状态** | 圆角矩形 |
| **转换（迁移）** | 带箭头的线，标注触发事件/条件/动作 |

#### 4.3.6 活动图（Activity Diagram）（★★）

**作用**：描述系统的工作流程，类似流程图。

**特殊元素**：

| 元素 | 含义 |
|------|------|
| **动作** | 圆角矩形 |
| **判定/分支** | 菱形 |
| **分叉/汇合** | 粗横线（表示并行活动的开始/结束） |
| **泳道** | 将活动按责任分组 |

> 🎯 **活动图 vs 状态图**：
> - 活动图：描述**工作流**（做什么，过程），面向活动
> - 状态图：描述**对象状态变迁**（什么状态），面向状态

#### 4.3.7 构件图（Component Diagram）与部署图（Deployment Diagram）

| 图 | 作用 |
|------|------|
| **构件图** | 描述软件构件（模块、库、可执行文件）及其依赖关系 |
| **部署图** | 描述硬件节点及其上运行的软件构件的物理架构 |

### 📝 本章典型真题

**【2023年真题改编】**
> 在 UML 用例图中，"取款"用例包含了"验证身份"用例，它们之间的关系是（）。

> 在 UML 类图中，类 A 的一个对象包含类 B 的一个对象，且类 B 的对象不能独立于类 A 存在，则 A 和 B 的关系是（）。
> A. 依赖　　B. 关联　　C. 聚合　　D. 组合

**答案：D**
**解析**：部分不能独立于整体存在，生命周期相同，这是组合关系。

**【2021年真题改编】**
> UML 中的（）图强调对象之间消息的时间顺序。
> A. 用例图　　B. 类图　　C. 顺序图　　D. 活动图

**答案：C**
**解析**：顺序图（序列图）的核心特征就是强调消息的时间顺序。

## 第五章 软件测试（★★★ 重点）

### 5.1 软件测试的目的与原则

**教材定义**：
> 软件测试是为了发现错误而执行程序的过程。一个好的测试用例是能够发现至今为止尚未发现的错误的测试用例。一次成功的测试是发现了至今为止尚未发现的错误的测试。

**软件测试原则**（教材）：
1. 所有测试都应追溯到用户需求
2. 应尽早和不断地进行软件测试（Pareto 原则：80% 的错误集中在 20% 的模块中）
3. 完全测试是不可能的，测试需要终止
4. 充分注意测试中的群集现象
5. 程序员应避免检查自己的程序
6. 尽量避免测试的随意性

### 5.2 测试方法分类（★★★ 核心考点）

#### 5.2.1 白盒测试（结构测试 / 逻辑驱动测试）

**教材定义**：
> 白盒测试是根据程序的内部结构和处理过程来设计测试用例，对程序的逻辑路径进行测试。

**白盒测试方法**（★★★ 必考）：

| 方法 | 说明 | 覆盖程度 |
|------|------|----------|
| **语句覆盖** | 每条语句至少执行一次 | 最弱 |
| **判定覆盖（分支覆盖）** | 每个判定的真假分支至少执行一次 | 弱 |
| **条件覆盖** | 每个判定中的每个条件的可能取值至少执行一次 | 中 |
| **判定/条件覆盖** | 同时满足判定覆盖和条件覆盖 | 中 |
| **条件组合覆盖** | 每个判定中条件的各种可能组合至少执行一次 | 强 |
| **路径覆盖** | 程序中每条可能的路径至少执行一次 | 最强 |

> 🎯 **覆盖强度排序**（必背）：
> 语句覆盖 < 判定覆盖 < 条件覆盖 < 判定/条件覆盖 < 条件组合覆盖 < 路径覆盖

**McCabe 复杂度（环路复杂度）**（★★ 常考计算）：

```
V(G) = m - n + 2
```
- m = 判定节点数（判断框数）
- n = 节点数
- 或：V(G) = 封闭区域数 + 1

> V(G) 的含义：程序中独立路径的条数，即至少需要 V(G) 个测试用例才能覆盖所有路径。

#### 5.2.2 黑盒测试（功能测试 / 数据驱动测试）

**教材定义**：
> 黑盒测试是在程序接口进行的测试，它只检查程序功能是否能按照规格说明书的规定正常使用。

**黑盒测试方法**（★★★ 必考）：

| 方法 | 说明 |
|------|------|
| **等价类划分** | 将输入数据划分为若干等价类，每个等价类选一个代表值 |
| **边界值分析** | 重点测试输入等价类的边界值 |
| **错误推测法** | 基于经验和直觉推测可能的错误 |
| **因果图法** | 用因果图描述输入条件的组合与输出的关系 |
| **判定表法** | 用表格列出所有条件组合和对应动作 |

**等价类划分**（★ 重点）：
- **有效等价类**：合理的输入数据
- **无效等价类**：不合理的输入数据
- 设计原则：一个测试用例应尽可能多地覆盖尚未被覆盖的有效等价类，一个测试用例只能覆盖一个无效等价类

### 5.3 测试阶段（★★★ 高频考点）

| 测试阶段 | 测试对象 | 主要方法 | 负责人员 |
|----------|----------|----------|----------|
| **单元测试** | 模块/函数 | 白盒测试 | 程序员 |
| **集成测试** | 模块间接口 | 黑盒为主，白盒为辅 | 测试人员 |
| **确认测试** | 软件需求 | 黑盒测试 | 测试人员 |
| **系统测试** | 整个系统（软硬件+数据） | 黑盒测试 | 独立测试组 |

**集成测试的两种策略**（★ 常考）：

| 策略 | 方式 | 特点 |
|------|------|------|
| **自顶向下** | 从主控模块开始，逐层向下集成 | 需要桩模块（stub），早期可看到整体 |
| **自底向上** | 从底层模块开始，逐层向上集成 | 需要驱动模块（driver），底层模块充分测试 |

**桩模块（Stub）与驱动模块（Driver）**（★ 常考）：
- **桩模块**：模拟被调用模块（自顶向下集成时使用）
- **驱动模块**：模拟调用模块（自底向上集成时使用）

### 5.4 测试相关概念

| 概念 | 定义 |
|------|------|
| **α 测试** | 在开发环境下，由用户在开发人员指导下进行的测试 |
| **β 测试** | 在实际使用环境下，由用户进行的测试 |
| **回归测试** | 修改错误后重新测试，确保修改未引入新的错误 |
| **验收测试** | 由用户进行的最终测试，确认系统满足需求 |

## 第六章 软件维护（★★ 常考）

### 6.1 软件维护的定义

**教材定义**：
> 软件维护是指在软件交付使用之后，为了改正错误或满足新的需要而修改软件的过程。

### 6.2 软件维护的分类（★★★ 必考）

| 类型 | 定义 | 举例 | 占比 |
|------|------|------|------|
| **改正性维护** | 诊断和改正错误 | 修复 Bug | 约 20% |
| **适应性维护** | 使软件适应环境变化 | 系统升级、数据库迁移 | 约 25% |
| **完善性维护** | 增加新功能或改进性能 | 添加报表功能 | 约 50%（最多） |
| **预防性维护** | 提高软件的可维护性和可靠性 | 重构代码、改善结构 | 约 5% |

> 🎯 **考试关键点**：
> - 完善性维护占比最大（约 50%）
> - 改正性维护 ≠ 完善性维护：前者修 Bug，后者加功能

### 6.3 软件可维护性

**决定软件可维护性的因素**（教材）：

| 因素 | 影响 |
|------|------|
| **可理解性** | 能否被维护人员理解 |
| **可测试性** | 能否被测试和验证 |
| **可修改性** | 能否被容易地修改 |
| **可移植性** | 能否移植到不同环境 |
| **可重用性** | 组件能否在其他系统中复用 |

### 6.4 软件再工程与逆向工程（★ 偶考）

| 概念 | 定义 |
|------|------|
| **逆向工程** | 从已有软件中提取设计信息（代码→设计→需求） |
| **正向工程** | 从设计信息生成代码（需求→设计→代码） |
| **再工程** | 对已有系统进行审查和改造，以新形式重建 |
| **重构** | 不改变外部行为的前提下改进内部结构 |

---

### 📝 本章典型真题

**【2022年真题改编】**
> 为了使软件适应运行环境的变化而修改软件的维护属于（）。
> A. 改正性维护　　B. 适应性维护　　C. 完善性维护　　D. 预防性维护

**答案：B**
**解析**：适应环境变化 = 适应性维护。

**【2019年真题改编】**
> 在各类软件维护活动中，占比最大的是（）。
> A. 改正性维护　　B. 适应性维护　　C. 完善性维护　　D. 预防性维护

**答案：C**
**解析**：完善性维护约占 50%，占比最大。

## 第七章 软件质量保证（★★ 常考）

### 7.1 软件质量的概念

**教材定义**：
> 软件质量是指反映软件系统或软件产品满足明确和隐含需求的能力的特性的总和。

### 7.2 软件质量特性模型

#### 7.2.1 McCall 质量模型（★★ 常考）

McCall 将软件质量因素分为三类：

| 类别 | 质量因素 |
|------|----------|
| **产品修正**（与适应变化有关） | 可维护性、可测试性、灵活性 |
| **产品转移**（与环境适应有关） | 可移植性、可重用性、互操作性 |
| **产品运行**（与运行有关） | 正确性、可靠性、效率、完整性、可用性 |

#### 7.2.2 ISO/IEC 9126 质量模型（★★★ 必考）

| 质量特性 | 子特性 |
|----------|--------|
| **功能性** | 适合性、准确性、互操作性、安全性、功能依从性 |
| **可靠性** | 成熟性、容错性、可恢复性 |
| **易用性** | 易理解性、易学性、易操作性 |
| **效率** | 时间特性、资源特性 |
| **可维护性** | 易分析性、易改变性、稳定性、可测试性 |
| **可移植性** | 适应性、易安装性、一致性、可替换性 |

> 🎯 **记忆口诀**：功（功能性）可（可靠性）用（易用性）效（效率）维（可维护性）移（可移植性）

### 7.3 软件评审（技术评审 / 代码走查）

**教材定义**：
> 软件评审是对软件元素或项目状态的一种评估手段，以确定其结果与必要的结果之间的差距。

**评审类型**：

| 类型 | 说明 |
|------|------|
| **设计评审** | 对概要设计和详细设计的审查 |
| **代码审查** | 对代码进行人工检查（不运行程序） |
| **代码走查** | 由小组逐步检查代码逻辑 |

### 7.4 软件容错技术（★ 偶考）

**教材定义**：
> 软件容错是指在系统出现故障的情况下，仍能提供规定服务的能力。

**容错方法**：
- **冗余**：N 版本程序设计、恢复块方法
- **恢复块方法**：主块执行，验证不通过则执行替换块

### 📝 本章典型真题

**【2021年真题改编】**
> ISO/IEC 9126 软件质量模型中，不包含以下哪个特性？
> A. 功能性　　B. 可靠性　　C. 可复用性　　D. 可维护性

**答案：C**
**解析**：ISO 9126 六大特性为：功能性、可靠性、易用性、效率、可维护性、可移植性。没有"可复用性"这一独立特性（它属于可维护性的子特性）。

## 第八章 软件过程改进（★★ 常考）

### 8.1 能力成熟度模型 CMM（★★★ 高频考点）

**教材定义**：
> CMM（Capability Maturity Model）是由美国卡内基梅隆大学软件工程研究所提出的，用于评价软件开发组织的软件过程能力成熟度的模型。它提供了一种评价软件承接方能力的方法，帮助软件组织改进其软件过程。

**CMM 五个成熟度等级**（★★★ 必背表格）：

| 等级 | 名称 | 关键特征 | 关键过程域 |
|------|------|----------|------------|
| **1** | **初始级** | 软件过程杂乱无章，成功依赖个人努力 | 无 |
| **2** | **可重复级** | 建立基本项目管理过程，有过程准则可重复成功 | 需求管理、项目计划、配置管理等 |
| **3** | **已定义级** | 软件过程文档化、标准化，所有项目采用标准过程 | 软件过程定义、培训、集成管理等 |
| **4** | **已管理级** | 制定了详细的度量标准，软件过程和产品质量可度量 | 定量过程管理、软件质量管理 |
| **5** | **优化级** | 通过定量反馈和新技术持续改进过程 | 缺陷预防、技术变更管理、过程变更管理 |

> 🎯 **记忆技巧**：
> 初始（混乱）→ 可重复（有管理）→ 已定义（有标准）→ 已管理（可度量）→ 优化（持续改进）
> 
> 关键区别：
> - 1→2：建立了项目管理
> - 2→3：过程标准化
> - 3→4：引入量化管理
> - 4→5：持续优化

### 8.2 CMMI（★★ 常考）

**教材定义**：
> CMMI（Capability Maturity Model Integration）是 CMM 的升级版本，将软件工程、系统工程等多个学科的过程改进集成在一起。

**CMMI 两种表示方法**：

| 表示法 | 说明 |
|--------|------|
| **连续式** | 按过程域评估能力等级（0–3 级） |
| **阶段式** | 按整体成熟度评估（1–5 级，类似 CMM） |

**CMMI 成熟度等级（阶段式）**：

| 等级 | 名称 |
|------|------|
| 1 | 初始 |
| 2 | 已管理 |
| 3 | 已定义 |
| 4 | 定量管理 |
| 5 | 优化 |

### 8.3 软件过程评估与能力评价

| 方法 | 目的 | 使用者 |
|------|------|--------|
| **软件过程评估** | 改进本组织的过程 | 组织内部 |
| **软件能力评价** | 评价承接方的能力 | 甲方/客户 |

## 第九章 软件项目管理（★★ 常考）

### 9.1 项目管理概述

**项目管理九大知识领域**（教材）：

| 领域 | 主要内容 |
|------|----------|
| 范围管理 | 确定项目包含什么和不包含什么 |
| 进度管理 | 编制进度计划、控制进度 |
| 费用管理 | 估算成本、控制成本 |
| 质量管理 | 质量计划、质量保证、质量控制 |
| 人力资源管理 | 组织团队、管理团队 |
| 沟通管理 | 信息分发、绩效报告 |
| 风险管理 | 风险识别、分析、应对 |
| 采购管理 | 外包和采购管理 |
| 整体管理 | 综合协调以上各方面 |

### 9.2 软件项目估算（★★★ 高频考点）

#### 9.2.1 项目估算的内容

| 内容 | 说明 |
|------|------|
| **规模估算** | 估算软件的大小 |
| **工作量估算** | 估算需要的人月/人天 |
| **成本估算** | 估算开发成本 |

#### 9.2.2 规模估算方法

**（1）LOC 估算法（代码行法）**

> 通过估算软件的源代码行数来度量软件规模。
> - 优点：直观、简单
> - 缺点：依赖编程语言、不适用于非过程语言

**（2）功能点（FP）估算法**

> FP（Function Point）是一种衡量工作量大小的单位，与编程语言无关。

**功能点计算**：
```
功能点 = 信息处理规模 × 技术复杂度因子
```

**信息处理规模包含五类**（★★★ 常考）：

| 类型 | 说明 |
|------|------|
| **外部输入数（EI）** | 向系统输入数据的事务 |
| **外部输出数（EO）** | 系统向外部输出的数据 |
| **外部查询数（EQ）** | 输入即输出的简单查询 |
| **内部逻辑文件数（ILF）** | 系统内部的逻辑文件 |
| **外部接口文件数（EIF）** | 与其他系统共享的接口文件 |

**技术复杂度因子**：
```
技术复杂度 = 0.65 + 0.01 × Σ(调节因子)
```

#### 9.2.3 工作量估算模型 COCOMO（★★★ 常考）

**教材定义**：
> COCOMO（Constructive Cost Model）是一种精确的、易于使用的工作量估算方法。

**COCOMO 的三个层次**：

| 层次 | 说明 |
|------|------|
| **基本 COCOMO** | 用代码行数的函数直接计算工作量 |
| **中间 COCOMO** | 在基本模型基础上，用产品、硬件、人员、项目等因素调整 |
| **详细 COCOMO** | 在中间模型基础上，分模块层、子系统层、系统层详细估算 |

**基本 COCOMO 公式**：
```
E = a × (KLOC)^b
D = c × (E)^d
```
- E = 工作量（人月）
- D = 开发时间（月）
- KLOC = 代码行数（千行）

**项目类型**（★ 常考）：

| 类型 | 特点 | a, b 值 |
|------|------|---------|
| **有机型** | 小团队、熟悉环境、需求灵活 | a=2.4, b=1.05 |
| **半分离型** | 中等规模、中等经验 | a=3.0, b=1.12 |
| **嵌入型** | 硬实时、严格约束、复杂接口 | a=3.6, b=1.20 |

#### 9.2.4 项目进度管理

**（1）甘特图（Gantt Chart / 横道图）**

**教材描述**：
> 甘特图以横线来表示每项活动的起止时间。

| 特点 | 说明 |
|------|------|
| 优点 | 简单、明了、直观、易于编制 |
| 缺点 | 不能表达活动之间的复杂关系，难以进行定量计算和优化 |

**（2）PERT 图 / 计划评审技术（★★★ 高频计算考点）**

**教材定义**：
> PERT 是利用网络分析制定计划以及对计划予以评价的技术。它能协调整个计划的各道工序，合理安排人力、物力、时间、资金，加速计划的完成。

**时间估算公式**（★★★ 必背）：
```
期望时间 ti = (ai + 4mi + bi) / 6
```
- ai = 乐观时间（最短）
- mi = 最可能时间
- bi = 悲观时间（最长）

**方差公式**：
```
σ²i = (bi - ai)² / 36
σi = (bi - ai) / 6
```

**（3）关键路径法 CPM（★★★ 高频考点）**

**核心概念**：

| 概念 | 定义 |
|------|------|
| **最早开始时间（ES）** | 某活动能够开始的最早时间 |
| **最早结束时间（EF）** | EF = ES + 工期 |
| **最迟结束时间（LF）** | 为按时完成，某活动必须完成的最迟时间 |
| **最迟开始时间（LS）** | LS = LF - 工期 |
| **关键活动** | ES = LS（或 EF = LF）的活动 |
| **关键路径** | 关键活动组成的路径，总持续时间最长 |
| **总时差** | LS - ES 或 LF - EF（不影响总工期的机动时间） |
| **自由时差** | 紧后工作的最早开始时间 - 本工作最早完成时间 |

> 🎯 **关键路径规则**：
> - 正推（前→后）：取所有前驱 EF 的最大值
> - 逆推（后→前）：取所有后继 LS 的最小值
> - 关键路径 = 总时差为 0 的活动链 = 最长路径

### 📝 本章典型真题

**【2023年真题改编 — PERT 计算】**
> 某软件项目的乐观时间为 6 周，最可能时间为 9 周，悲观时间为 18 周，则期望时间为（）周。
> A. 9　　B. 10　　C. 11　　D. 12

**答案：B**
**解析**：t = (6 + 4×9 + 18) / 6 = (6 + 36 + 18) / 6 = 60 / 6 = 10

**【2022年真题改编 — COCOMO】**
> 在 COCOMO 模型中，项目类型分为有机型、半分离型和嵌入型，其中（）的开发难度最大。
> A. 有机型　　B. 半分离型　　C. 嵌入型　　D. 三者相同

**答案：C**
**解析**：嵌入型的系数 a=3.6、b=1.20 最大，表示最复杂、工作量最大。

**【2021年真题改编 — 配置管理】**
> 在软件配置管理中，（）是经过正式评审和批准的配置项的集合。
> A. 配置库　　B. 基线　　C. 版本　　D. 快照

**答案：B**
**解析**：基线是经过正式评审和批准的配置项集合，是后续开发的基准。

**【2020年真题改编 — 进度管理】**
> 甘特图不能反映的信息是（）。
> A. 各项活动的起止时间
> B. 各活动之间的先后依赖关系
> C. 活动的进度
> D. 项目的开始和结束时间

**答案：B**
**解析**：甘特图的缺点就是不能系统地表达各活动之间的复杂依赖关系。

## 附录 C：必背公式与口诀

### 公式

```
McCabe 环路复杂度 V(G) = m - n + 2  （m=判定节点数，n=总节点数）
                或 V(G) = 封闭区域数 + 1

PERT 期望时间   ti = (ai + 4mi + bi) / 6
PERT 方差       σ² = (bi - ai)² / 36

COCOMO 工作量   E = a × (KLOC)^b
COCOMO 开发时间 D = c × (E)^d

功能点 FP = 信息处理规模 × (0.65 + 0.01 × Σ调节因子)
```

### 口诀

**耦合强度排序**：非→数→标→控→公→内
（非直接→数据→标记→控制→公共→内容）

**内聚强度排序**：偶→逻→时→过→通→顺→功
（偶然→逻辑→时间→过程→通信→顺序→功能）

**CMM 五级**：初→可→已定→已管→优
（初始→可重复→已定义→已管理→优化）

**软件维护分类占比**：完（50%）>适（25%）>改（20%）>预（5%）

## 附录 D：下午题案例分析攻略

### 数据流图（DFD）题（15 分）

**题型固定**：给出不完整的 DFD，要求补充外部实体、数据存储、加工名、缺失数据流。

**解题步骤**：
1. 先读题目文字描述，提取所有外部实体、加工、数据存储、数据流名称
2. 对照 DFD，找出缺失的元素
3. 检查父图子图平衡（输入输出数据流是否一致）
4. 检查数据守恒（加工的输出必须由输入产生）
5. 确认外部实体不与数据存储直接交互

**常见扣分点**：
- 数据流名称与题目描述不一致
- 遗漏了数据流
- 父图子图不平衡
- 加工没有输出（或没有输入）

### UML 建模题（15 分）

**题型**：给出一段系统描述，要求补全用例图或类图。

**用例图技巧**：
- 先找出所有 Actor（人员角色、外部系统）
- 再找功能需求（每个功能 = 一个用例）
- 注意包含（include）和扩展（extend）关系的方向
- 包含：箭头指向被包含用例（无条件必须执行）
- 扩展：箭头指向被扩展用例（有条件才执行）

**类图技巧**：
- 先找出所有名词（可能的类）
- 再找动词（可能的方法/关联）
- 注意类之间的关系类型和多重度
- 聚合 vs 组合：部分能否独立存在

### 设计模式题（15 分）

**题型**：给出一段代码或类图，识别设计模式，填空补全。

**常考设计模式**（★★★ 必须掌握）：

| 模式 | 类型 | 核心思想 |
|------|------|----------|
| **工厂方法** | 创建型 | 定义创建对象的接口，由子类决定实例化哪个类 |
| **抽象工厂** | 创建型 | 提供创建一系列相关对象的接口 |
| **单例模式** | 创建型 | 确保一个类只有一个实例 |
| **适配器模式** | 结构型 | 将一个类的接口转换为客户期望的接口 |
| **桥接模式** | 结构型 | 将抽象与实现分离 |
| **装饰器模式** | 结构型 | 动态地为对象添加职责 |
| **外观模式** | 结构型 | 为子系统提供统一接口 |
| **代理模式** | 结构型 | 控制对对象的访问 |
| **观察者模式** | 行为型 | 对象间一对多依赖，状态变化时通知所有依赖者 |
| **策略模式** | 行为型 | 定义一系列算法，封装起来可互换 |
| **命令模式** | 行为型 | 将请求封装为对象 |
| **状态模式** | 行为型 | 允许对象在其内部状态改变时改变行为 |
| **模板方法** | 行为型 | 定义算法骨架，延迟某些步骤到子类 |

## 附录 E：结构化开发 vs 面向对象 对比总结

| 对比项 | 结构化开发 | 面向对象开发 |
|--------|-----------|-------------|
| 核心思想 | 自顶向下、逐步分解功能 | 以对象为核心，封装、继承、多态 |

| 复用性 | 模块级复用，较低 | 类/构件级复用，较高 |
| 适用场景 | 数据处理、算法密集 | 复杂业务系统、GUI 应用 |
| 代表模型 | 瀑布模型 | 喷泉模型、RUP |
