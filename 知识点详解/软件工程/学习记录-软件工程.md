软件开发模型
    瀑布模型
        最早的模型，将开发过程分为若干个 线性顺序 的阶段
        需求分析 - 软件设计 - 编码 - 测试 - 运行维护
        核心规则
            每个阶段必须在前一个阶段完成后才能开始
            每个阶段完成后，会有对应的文档生成，也就是里程碑
            阶段间由顺序性和依赖性
            强调文档驱动，每个阶段由明确的输入输出
        适用场景：需求明确，变动少的，技术成熟，变动小的，强调文档，有利于项目质量和跟踪
        缺点：不够灵活，不适用需求变化，用户需要在后期才能看到软件，反馈滞后，风险往往在后期暴露，改动代价大
        瀑布模型最大的缺点就是不适应需求变化，瀑布模型不允许回溯
        瀑布模型不是迭代的，是顺序的
        V瀑布是瀑布模型的变体，强调测试与开发阶段的关系
    增量模型
        融合了瀑布模型的基本成分和原型实现的迭代特征
        核心规则
            将软件产品分解为一系列增量构件
            每个增量都是一个可运行，可交付的产品子集
            每个增量都经历 需求 设计 编码 测试的完整过程
            第一个增量通常是核心功能，后续逐步添加增量功能
        关键点
            每一轮增量开发都使用瀑布模型
            每一轮增量都会产生一个可用版本
            允许在开发过程中调整后续增量计划
        优点：降低风险，用户能早期获得部分功能并提供反馈，有利于后续人员灵活分配
        缺点：需要有良好的 软件体系结构设计，否则增量集成困难，增量的划分需要经验
        增量模型是瀑布模型+迭代，每次增量都能交付
        第一个增量通常是核心
        增量模型特别适合大型项目，可以分而治之
        增量模型结合了 瀑布模型 + 演化模型的迭代优点
        增量模型支持并行开发
    螺旋模型
        其核心是将开发过程组织为一系列螺旋形的迭代，每个迭代包含四个象限：制定计划（目标设定） ， 风险分析（识别和消除风险） ， 实施工程（开发和验证） ， 客户评价（评价与反馈）
        特点
            强调 风险分析 ，每轮迭代都进行风险评估
            融合了瀑布模型（系统性）和原型模型（迭代性）的优点
            适用于 大型，高风险的项目
            采用原型或者其他方式来降低风险
        适合：大规模内部开发，高风险项目，需求不明确，且存在技术风险的项目
        螺旋模型的核心特点是 风险驱动 
        要记住四象限的顺序：制定计划 风险分析 实施工程 客户评价
        螺旋模型适用大场景 高风险
    快速原型模型
        快速构建一个原型，让用户试用并反馈，然后根据反馈修改和完善原型，直到满足用户需求
        分抛弃型原型（仅用于获取需求，验证后被抛弃，按照瀑布模型重新开发）演化型原型（原型不断演化，最终演化为成品）
        特点：
            适用于需求不明确的项目
            以用户需求为驱动
            用户能在早期看到软件的外观和功能
            原型可以是界面原型或功能原型
        优点：
            有助于获取和验证需求，用户早期参加，避免误解，降低项目风险
        缺点：
            原型质量可能不高，直接演化为产品可能会有缺陷，可能会导致开发者忽视体系结构设计
        原型模式的驱动是 用户需求
        适用需求不明确的场景
    
    V模型
        V模型是瀑布模型的一种变体，核心思想是强调 测试与开发阶段的对应关系
        需求分析                        验收测试
            概要设计                系统测试
                详细设计        集成测试
                    编码    单元测试
        左侧是开发阶段，自顶向下；右侧是测试阶段，自底向上，形成v字
        特点：
            强调测试的重要性，测试贯穿整个开发过程
            每个开发阶段都有对应的测试阶段
            是瀑布模型的扩展，仍然是线性模型
        需记住每个开发阶段对应的测试阶段，常考核
        V模型不是编码后才做测试，而是每个开发阶段都有对应的测试阶段

    
    喷泉模型
        喷泉模型是一种以用户需求为驱动，以面向对象技术为基础的开发模型
        特点
            开发过程是迭代的，无间隙的
            各个阶段之间没有明显的界限
            支持面向对象的开发方法
            阶段可以反复进行，各个阶段互相重叠，反复迭代
    

    敏捷开发方法
        敏捷开发是一种 以人为核心，迭代，循序渐进的开发方法，四个核心价值观
            个体和互动 高于 流程和工具
            工作的软件 高于 详尽的文档
            客户合作  高于 合同谈判
            响应变化  高于 遵循计划
        主要方法：
            极限编程：结对编程，持续集成，测试驱动开发，简单设计，小型发布，代码共有，强调技术实现和高质量代码
        角色：产品负责人，Scrum Master服务型领导，,开发团队 
        活动：冲刺，计划会，每日站会，评审会，回顾会
        工件：产品待办列表，冲刺待办列表，增量
        强调管理实践和团队协作
    
    统一过程 RUP
        统一过程是一种以用例驱动，以体系结构为中心，迭代和增量的软件开发过程
        阶段
            初始阶段 ：建立项目的业务模型和范围
            细化阶段：建立结构体系基线，消除高风险因素
            构造阶段：开发所有剩余构建，组装成成品
            移交阶段：将产品交付给用户
        特点
            用例驱动，以体系结构为中心
            迭代和增量 ，每次迭代都包含完整的开发工作流 需求，分析，设计，实现，测试
        统一过程是过程框架，可以裁剪为敏捷，但是本身不是典型的敏捷
        细化阶段：建立结构体系基线，消除高风险
        统一过程是用例驱动，体系结构化，迭代和增量
    

    构建化开发模型和复用
        结构化开发：是指复用已有的结构组成软件系统的操作方法
        核心
            构件：可复用的软件单元，具有独立的功能，通过接口与其他构件交互
            软件复用：重复使用已有的软件构件来组成新系统
            构件库：存放可复用构件的仓库
        复用层次
            代码复用：直接复用原代码或目标代码
            设计复用：复用设计方案和设计模式
            分析复用：复用分析方法和分析结果
            测试信息复用：复用测试用例和测试数据
        优点：
            提高开发效率，提高软件质量，降低开发成本

    逆向工程
        是指通过分析已有系统，从其代码或实现方式中提取设计和需求信息，是正向工程的逆过程
        四个抽象层次
            实现级：程序的语法结构，AST，原代码
            结构级：程序的分量及其关系，类，函数调用图
            功能级：程序的功能与交互关系 数据流和控制流
            领域级：与领域相关的概念和关系，业务规则
        逆向工程不是破解软件，是从已有软件中分析出设计和需求
        重构是在同一层级上的变换
        再工程：对已有系统进行审查和改造



软件需求分析
    需求分析过程
        需求分析是软件开发过程中最重要的阶段之一，他的任务是明确，准确回答系统需要做什么
        需求分析的任务
            确定系统的综合要求：功能需求，性能需求，可靠性和可用性需求，出错处理需求，接口需求，约束条件，逆向需求等
            分析系统的数据要求：建立数据模型（ER图）
            导出系统的逻辑模型：数据流图和数据字典
            修正系统的开发计划
            开发原型系统
        需求分析的过程
            需求获取：通过个各种方法收集需求
            需求分析与协商：分析需求的合理性和一致性
            需求规格说明：编写需求规格说明书SRS
            需求验证：验证需求的完整性与正确性
            需求管理：跟踪和管理需求变更
        需求分析不是在写代码，是写代码之前明确干什么
        需求分析不止关注功能需求，还有非功能需求，比如性能需求，约束条件等
    需求的分类
        功能需求：系统必须完成的需求或者功能
        非功能需求：系统必须满足的约束条件
            性能需求
            安全性需求：数据加密，权限控制
            可靠性需求：系统可用性，故障恢复
            可维护性需求：模块化，可读性
            可移植性需求：跨平台运行能力
        约束设计：限制条件
            开发语言限制
            运行平台限制
            法规和标准限制
            成本和时间限制




数据流图
    是结构化分析方法中描述系统的 数据护理过程 的图形工具
    四个基本要素
        外部实体：系统的数据来源或者去处，用户   其它系统
        加工/处理：对数据进行处理的功能，验证密码
        数据存储：数据存储的地方，数据库 文件
        数据流：数据的流动方向
        外部实体<->加工
        加工<-> 加工
        加工<-> 数据存储
        其他的都是违法的 外部实体 ↔ 外部实体  外部实体 ↔ 数据存储  数据存储 ↔ 数据存储
    黑洞：只有输入数据流，没有输出数据流，不符合逻辑
    白洞（奇迹）：只有输出数据流，没有输入数据流，数据凭空产生，不符合逻辑
    灰洞：输入数据不足以支撑输出数据，就给了用户id，没查数据库就输出了用户信息
    分层结构
        复杂的系统功能无法在一个图中展示出来，因此DPD采用 自顶向下，逐层分解的结构，同时衍生出最重要的，父子平衡关系
        顶层图：又叫上下文图
            只有一个加工，代表整个系统，只有外部实体和系统之间的数据流，无数据存储，作用是定义系统边界，明确系统和外界的交互范围
        0层图
            将顶层图中的 整个系统加工 拆解为多个核心功能加工
            新增数据存储，描述系统核心模块之间的数据流动
            是顶层图的子图，顶层图是他的父图
        1层及以下子图
            将0层图中的某一个加工进一步拆解为更细的子加工
    父图子图平衡原则
        子图的所有输入数据流和输出数据流，都必须和父图中对应加工的输入输出在数量和名称上完全一致
    数据字典：数据字典是对数据流图中所有元素的详细定义和说明，是DFD的配套说明书，数据的具体组成，格式，类型都在数据字典中定义
        核心条目
            数据流条目：定义数据的组成，来源，去向，流量，数据项组成；用户信息=用户ID+姓名+手机号
            数据存储条目：定义数据存储的结构，主键，存取方式
            数据项条目：最小数据单元，不可拆分，数据项的类型，长度，取值范围；用户id= 18位字符串
            加工条目：定义加工的处理逻辑，输入输出，处理规则，触发条件，加工的结构化语言描述
        DFD中出现的所有元素，都必须在数据字典中有对应定义
        - `=` 表示"由……组成"
        - `+` 表示"与"
        - `[ | ]` 表示"或"
        - `{ }` 表示"重复"
        - `( )` 表示"可选"
        学生成绩 = 学号 + 姓名 + {课程名 + 成绩}
        学号 = 10位数字
        课程名 = 1~20个字符
    加工逻辑方法
        加工逻辑用于描述DFD中的 加工 的内部逻辑，主要描述方法有三种
        结构化下语言：介于自然语言和编程语言之间，主要是顺序，选择（if_then_else）,循环（FOR/WHILE）。特点是比自然语言精准，比编程语言易读性强
            IF 会员等级为钻石会员 THEN
                折扣率 = 0.8
            ELSE IF 会员等级为黄金会员 AND 累计消费满1000元 THEN
                折扣率 = 0.85
            ELSE IF 会员等级为黄金会员 AND 累计消费不满1000元 THEN
                折扣率 = 0.9
            ELSE
                折扣率 = 0.95
            ENDIF
            订单实付金额 = 订单原价 × 折扣率
        判定表：用于描述加工中多个条件组合和对应执行动作之间的关系，能完全覆盖所有场景的条件组合
            条件桩：列出所有需要判断的条件，所有需要判断的问题项
            条件项：列出所有条件的取值范围
            动作桩：满足条件后需要做的事情
            动作项：对应每个条件项，标注该条件项组合情况下要做的动作

            条件桩 / 动作桩        规则 1  规则 2  规则 3  规则 4  规则 5
            条件 1：钻石会员       是      否      否      否      否
            条件 2：黄金会员       否      是      是      否      否
            条件 3：累计消 满1000元 -      是      否      -      -
            动作 1：折扣率 0.8     √                                
            动作 2：折扣率 0.85            √                        
            动作 3：折扣率 0.9                    √                
            动作 4：折扣率 0.95                            √        
            适合：加工条件包含多个相互关联的条件，大量条件组合，每个条件对应不同的动作的场景

        判定树：判定树是判定表的图形化表示方式
            会员折扣判定
            ├─ 钻石会员 → 折扣率0.8
            └─ 非钻石会员
            ├─ 黄金会员
            │  ├─ 累计消费满1000元 → 折扣率0.85
            │  └─ 累计消费不满1000元 → 折扣率0.9
            └─ 普通会员 → 折扣率0.95
            适合：加工条件的逻辑有明确的层级关系，分支数量适中，需要向非技术用户展示逻辑的场景
        结构化语言：逻辑严谨，容易转换为代码，适合简单分支的加工
        判定表：覆盖所有条件组合，逻辑严谨，适合多条件多组合，无明显层级的
        判定树：直观易读，适合用户沟通，适合分支适中，有明确分层的场景
        动作桩：列出加工中可能出现的动作和操作
    需求验证
        核心目标是确保 软件需求规格说明书SRS（包括DFD）的正确性，完整性。可验证性，一致性，消除需求错误，遗漏，矛盾。防止后续开发阶段出现大规模的返工
        围绕以下八个核心
            正确性：要求需求必须真实，DFD是否包含了所有的用户业务功能，数据流是否符合流转规则，外部实体是否准确
            完整性
            一致性
            无二义性
            可验证性
            可修改性
            可追踪性
            可理解性
        需求验证方法：
            需求评审：非正式评审，正式评审
            原型法：用户不明确，无法准确描述业务场景，可以先搭建一个快速原型，让用户检查
            自动化辅助工具验证：通过结构化分析工具，自动校验 DFD 的父图子图平衡、数据流合法性、加工的黑洞 / 奇迹错误、数据字典与 DFD 的一致性，提升验证效率。
        需求的可实现性不在需求验证阶段，在设计阶段评估
    DFD的核心意义：结构化分析的核心工具，用于需求分析阶段，构建系统的功能逻辑模型
    数据流图不包含控制流
