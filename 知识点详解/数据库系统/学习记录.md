三级模式结构
    最重要的结构之一
    外模式:也叫用户模式，每个用户看到的数据可能不同，可以有多个外模式，，外模式是模式的一部分
    模式：逻辑模式，是所有数据的全局逻辑结构，描述了所有数据的逻辑结构和特征，是所有用户的公共数据视图，一个数据库只有一个模式
    内模式：数据的物理存储层面的结构，存储方式和索引组织方式，一个数据库只有一个内模式
    两级映像
        外模式/模式保证数据逻辑独立性，模式修改的时候，通过修改映射，外模式可以不变
        模式/内模式保证数据物理独立性，内模式修改的时候，通过修改映射，模式可以不变
    三级模式结构中，模式 描述了数据的，全局逻辑结构

关系模型
    关系模型，就是关系型数据库使用的模型，如oracle和mysql等
    关系：整个表：一张表
    元组：表格中的一行：一条记录
    属性：表格中的一列，一个而字段
    域：属性的取值范围
    主键：能够唯一标识一个元组的属性或属性组合
    外键：一个表中引用另一个表中的主键，用于表与表之间的关联
    关系的性质
        表中每一列是同质的，也就是同一列的数据类型相同
        每列必须有不同的属性名，可以出自同一个域
        列的顺序无所谓，行的顺序也无所谓
        每个分量都是一个不可再分的原子值，满足1NF
    关系的三类完整性约束
        实体完整性：主键值不能取空，否则无法唯一标识元组
        参照完整性：外键要么是空值，要么是另一个表中的某个主键值，防止引用了不存在的数据
        用户自定义完整性

关系代数
    关系代数是对关系表进行操作的数学语言
    选择：σ 西格玛：挑选符合条件的行
        示例：σ_{年龄>20}(学生) → 从学生表中找出年龄大于20的所有学生
    投影： π 读pi  挑选指定的列
        示例：π_{姓名,院系}(学生) → 从学生表中只取姓名和院系两列
        投影操作可能会去掉重复的行，因为投影得到的是集合，不允许重复
    并：∪ 两张表合并
        将两个关系表中的行合并，两个关系表的属性数量和类型需要是相同的
    差： - 从一张表中去除另一个表的内容
        R-S 表示在R中 不在S中的内容
    笛卡尔积： × 所有可能的组合
        R × S 表示R的每一行和S的每一行进行组合
        如果R有m行n列，S有p行q列，则R×S有m*p行n+q列
    连接： ⋈
        自然连接：对两个表，用相同的列名进行关联，组合所有列名的结果，去重重复结果
        等值连接：与自然连接一样，但是不去重
    除：÷
        R÷S 代表 找出R中 与 S所有元组都能关联的哪些元组
        > 示例：找出选修了课程号为"C01"和"C02"所有课程的学生
        > R÷S，其中R是选课表（学号, 课程号），S是{C01, C02}

关系代数表达式的等价变换
    选择和投影可以提前执行，减少中间结果的大小，是数据库查询优化的基本原理
    常用规则
        选择下推：尽早执行选择操作
        投影下推：尽早执行投影操作
        选择和笛卡尔积结合为连接
        选择的分配律:σ_{条件}(R⋈S) ≡ σ_{条件}(R)⋈S（当条件只涉及R的属性时）
    核心：先做选择，减少行数；再做投影，减少列数，最后连接，组合数据
    > 以下关系代数表达式中，与σ_{C1 AND C2}(R)等价的是（ C ）。
        > A. σ_{C1}(R) ∪ σ_{C2}(R)
        > B. σ_{C1}(R) ∩ σ_{C2}(R)
        > C. σ_{C1}(σ_{C2}(R))
        > D. σ_{C1}(R) × σ_{C2}(R)
关系代数与sql的关系
    选择 σ = where
    投影 π = select指定列名
    并 ∪ = UNION
    差 - =  EXCEPT / MINUS
    笛卡尔积 × = from 表1，表2 无条件连接
    自然连接 ⋈ = JOIN ON

关系演算
    关系演算与关系代数是关系操作的两种不同表达方式，三者表达能力等价
    关系元组演算
        以元组为变量，用谓词公式描述查询条件
        { t | 学生(t) ∧ t.年龄 > 20 } ：查询年龄大于20的学生
    域关系演算
        以属性值为变量，用谓词公式描述查询条件
        { <sname> | ∃sno, sage, sdept (学生(sno, sname, sage, sdept) ∧ sage > 20) } ：查询年龄大于20的学生姓名
    
    关系代数：过程化语言，描述怎么做
    元组关系演算：非过程化语言，描述要什么，用条件描述结构
    域关系演算：非过程化语言，以域为单位，QBE语言基础
    三者 表达能力等价，任何一种方式表达的查询都能用另外两种方式等价表达
    sql语言的本质是基于关系代数和关系演算的结合体

    索引
        创建：CREATE INDEX idx_name on 学生(姓名)
        删除：DROP INDEX idx_name

        类型
            聚簇索引：数据物理存储和索引顺序一致，一个表只能有一个

        DCL用于控制数据库的访问权限

    关系数据库规范化治理
        函数依赖：是描写属性之间约束关系的概念
            如果直到A的值，就能唯一确定B的值，就说明，B函数依赖于A，记作 A -> B,比如知道学号知道姓名，学号→姓名 是一个函数依赖
        三种函数依赖
            完全函数依赖
                如果 A→B，且A的任意一个真子集都无法唯一确定B，则称为B完全依赖于A；就如同（学号，课程号）→成绩，单独的学号或者课程号都无法确定成绩，所以成绩完全依赖于（学号，课程号）
            部分函数依赖
                如果A→B，且A的某个真子集也能决定B，则称为B不分函数依赖于A；(学号, 课程号)→姓名，但单独的学号就能决定姓名，所以姓名**部分函数依赖**于(学号, 课程号)
            传递函数依赖
                如果A→B，B→C，且B不能决定A，则说明C传递依赖于A
        部分函数依赖的前提是组合属性，如果是单一属性，就不存在部分函数依赖
        传递依赖要求 B不能决定A，如果B→A，那么传递依赖就不成立

    键与候选键，主键，外键
        候选键：能够唯一标识一个元组（一行数据）的最小的属性集合
            唯一性+最小性
        主键：从候选键中选择一个出来，作为表的主标识
        外键：一个表引用另一个表的主键的属性
        超键：能够唯一标识元组的组合，不要求最小，候选键是超键的子集
        关系R(A,B,C,D)，函数依赖集F={A→B, B→C, A→D}，看函数依赖中，只出现在最左边的，这些属性一定在候选键中；只出现在右边的，一定不在候选键中;左右都有的，可能在候选键中
        举例R(A,B,C,D), F={AB→C, C→D, D→B}
            > - A只在左边 → 一定在候选键中
            > - B：左边+右边 → 可能在候选键中
            > - C：左边+右边 → 可能在候选键中
            > - D：左边+右边 → 可能在候选键中
            > - 先试{A}⁺ = {A} → 推不出全部
            > - 试{A,B}⁺ = {A,B,C,D} ✓ → 但要检查是否最小
            >   - 去掉B：{A}⁺ = {A} ✗ → AB是候选键
            > - 试{A,C}⁺ = {A,C,D,B} ✓ → 去掉C：{A}⁺ = {A} ✗ → AC是候选键
            > - 试{A,D}⁺ = {A,D,B,C} ✓ → 去掉D：{A}⁺ = {A} ✗ → AD是候选键
            > - **候选键有3个：AB、AC、AD**
    
    范式理论
        范式是衡量关系模式好坏的标准，范式越高，数据冗余越少，但是查询可能越复杂
        第一范式：1NF
            表中每个属性都是不可再分的原子值，满足1NF是基本要求
        第二范式：2NF
            在1NF的基础上消除非主属性对候选键的部分函数依赖，也就是每个非主属性都完全依赖于候选键
            如果候选键是单个属性，自动满足2NF
        第三范式：3NF
            在2NF的基础上上，消除非主属性对候选键的传递函数依赖，也就是每个非主属性都不传递依赖于候选键
        BC范式：BCNF
            在1NF的基础上，每个非平凡函数依赖的决定因素都是超键，比3NF更加严格，不仅非主属性不能部分函数/传递依赖，主属性也不能
        范式等级 
            1NF ⊂ 2NF ⊂ 3NF ⊂ BCNF（越往右要求越严格）
        > 关系模式R(A,B,C,D)，函数依赖集F={AB→C, AB→D, C→B}，则R的候选键是（ ），最高范式是（ ）。
        >
        > **答案**：候选键为AB和AC，最高范式为2NF

    模式分解与规范化
        无损分解：分解后的关系通过自然连接能够恢复原来的关系，则该分解是无损分解
        保持函数依赖：分解后所有函数依赖，都能由分解后的各关系的函数依赖集推导出来

        简化判定无损分解（两分解情况）
            将R分解为R1和R2的时候
                1.先算R1 ∩ R2
                2.再算R1 - R2 和 R2 - R1
                3.算是否有满足R1 ∩ R2 → R1 - R2 或者 R1 ∩ R2 → R2 - R1,只要有一个满足的，就是无损分解 

    公理系统
        公理系统是推理函数依赖的基本规则
        自反律：如果Y ⊆ X，则X→Y
        增广率：如果X→Y，则XZ→YZ
        传递率：如果X→Y，Y→Z，则X→Z
        合并规则：如果X→Y，x→Z 则X→YZ
        分解规则：X→YZ → X→Y, X→Z
        伪传递规则：X→Y, WY→Z → WX→Z
    
    反规范化技术
        规范化减少了冗余，但是可能会影响查询效率，适当引入冗余技术
        常见手段
            增加派生列：把可以通过计算的结果存储为列
            增加冗余列：将其他表中的常用的列复制过来
            合并表：把需要经常连表查询的表合并为一张
            分割表：把大表按行或者按列拆分，减少扫描范围
    
数据库的控制功能
    事务的概念和ACID特性
        事务：数据库中一组不可拆分的操作序列，要么全成功，要么全不执行（回滚）
    ACID四个特性
        原子性：事务中的操作要么全做，要么全不做
        一致性：事务执行前后，数据库从一个一致状态到另一个一致状态
        隔离性：多个事务并发执行的时候，互不影响
        持久性：事务一旦提交是永久性的
    
    并发控制
        多个事务同时操作数据库的时候可能出现的问题
            丢失更新：两个事务同时修改一个数据，会有一个修改被覆盖
            不可重复读：事务内两次读取数据是不同的数据
            读脏数据：读到了其他事务未提交的数据
        封锁技术
            排他锁：X锁/写锁,事务对数据加X锁后，其他事务不能对他进行读或写，与其他锁不兼容
            共享锁：S锁/读锁，事务对数据加S锁后，其他事务只能读，不能写，兼容S锁，不兼容X锁
        两阶段封锁协议
            增长阶段：事务只能获取锁，不能释放锁
            收缩阶段：事务只能释放锁，不能获取锁
            两阶段封锁协议能够保证 可串行化调度 ， 也就是并发执行的结果等价于串行执行的结果
        死锁
            两个事务互相等待对方释放锁
        死锁处理
            预防：一次性申请所有锁，顺序加锁法（按照固定顺序加锁）
            检测与恢复：维护等待图，检测到环路就是存在死锁，选择一个事务作为牺牲者回滚

数据库恢复技术
    故障类型
        事务内故障：运算溢出，违反约束等，直接回滚即可
        系统故障：断电，系统崩溃，用日志+已提交事务
        介质故障：磁盘损坏，用备份+日志恢复
    日志技术
        日志记录了事务的所有操作，包括修改前和修改后
        UNDO：利用日志中修改前的值恢复数据
        REDO：利用日志中修改后的值重新执行操作
    检查点技术
        检查点是数据库系统的某个时间点的快照，记录了当时所有活跃的事务的状态
        有了检查点，恢复的时候不用扫描整个日志，只需要从最近的检查点开始
    系统恢复的步骤
        从日志最后反向查找，找出故障时所有活跃的事务，也就是未提交的事务，这步需要UNDO
        从检查点开始，向后扫描，找出所有在检查点后已提交，但是未写入磁盘的事务，这步需要RENO
    记忆口诀
        未完成的回滚（UNDO），已完成的重做（REDO）

数据库设计
    数据库设计主要分为六个阶段
        需求分析：了解用户需求，收集数据，产出：数据流图和数据字典
        概念结构设计：建立概念模型，产出ER图
        逻辑结构设计：将ER图转为关系模型，产出关系模型也就表结构
        物理结构设计：确定存储结构和索引，产出物理存储方案
        数据库实施：创建数据库，加载数据，产出的是运行的数据库
        数据库运行维护：性能调优，备份恢复，产物的是维护文档
    
    概念结构设计
        ER图设计方法
            自底向上：先设计各局部的ER图，再合并为全局ER图
            自顶向下：先设计全局框架，再逐步细化
        ER图合并时的冲突处理
            属性冲突：同一属性在不同的局部ER图中的类型或者取值范围不同
            命名冲突：同名异议或者异名同义
            结构冲突：同一实体在不同局部ER图中的属性不同，或者联系类型不同
    逻辑结构设计
        ER图转关系模式的规则
            一对一：合并到任意一方
            一对多：合并到多的一方
            多对多：独立建表
        三元联系的转换规则
            1：1：1   合并到任意一方
            1：1：n   合并到n方，加入另外一方的主键
            1：m：n   独立建表，包含三方主键和联系属性
            m:n:q    独立建表
    物理结构设计
        确定数据的存储结构，索引策略，分区策略
    
数据库新技术
    分布式数据库
        分布透明性的三个级别
            分片透明：最高级，用户不需要知道数据是如何分片的，用的时候当整体用即可
            位置透明：用户知道分片，但不知道具体哪个场地
            局部映像透明：用户知道数据存储在哪个场地，但是不用知道具体数据库管理系统DBMS的细节
    Nosql
        非关系型数据库
        适用于大数据高并发，灵活数据模型的场景
        类型：键值对（redis），文档型（MongoDb），列族（Hbase），图数据库

    数据仓库与数据挖掘
        数据仓库：面向主题，集成的相对稳定的，反应历史变化的数据集合，用于支持管理决策
        数据挖掘：从大量的数据中发现有用的知识和模式

    分布式数据库需要网络连接来连接各场地
