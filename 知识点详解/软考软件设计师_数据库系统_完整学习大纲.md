# 第一章 数据库系统概述

---

## 1.1 数据库系统基本概念

### 1.1.1 数据（Data）

**教材定义**：数据是描述事物的符号记录，是数据库中存储的基本对象。数据不仅包括数字，还包括文字、图形、图像、声音、视频等。

**通俗解释**：数据就是记录信息的符号。比如你的姓名"张三"、年龄"25"、照片，这些都是数据。数据本身可能没有意义，但经过解释（语义）就有了含义。

**详细说明**：在数据库中，数据和它的语义是不可分割的。例如，"张三"这个数据，结合"学生姓名"这个语义，才具有明确的含义。数据有"型"和"值"之分——"型"是指数据的结构，如学生(学号, 姓名, 年龄)；"值"是指具体的值，如('001', '张三', 20)。

### 1.1.2 数据库（DB, Database）

**教材定义**：数据库是长期储存在计算机内的、有组织的、可共享的数据集合。数据库中的数据按一定的数据模型组织、描述和储存，具有较小的冗余度、较高的数据独立性和易扩展性，并可为各种用户共享。

**通俗解释**：数据库就是一个"数据仓库"，按照一定规则把数据整齐地放好，方便查找和使用，不会有多余的重复数据，大家都来取用。

**详细说明**：数据库具有以下特点：(1) 数据结构化——数据库中的数据不再是孤立的，而是有组织的；(2) 数据的共享性高、冗余度低——数据可以被多个用户和应用共享，减少重复；(3) 数据独立性高——数据的逻辑结构和物理存储可以分开；(4) 由DBMS统一管理和控制——保证数据的安全性、完整性和并发控制。

### 1.1.3 数据库管理系统（DBMS, Database Management System）

**教材定义**：DBMS是位于用户与操作系统之间的一层数据管理软件，是系统软件。它的主要功能包括：数据定义功能、数据组织存储和管理、数据操纵功能、数据库的事务管理和运行管理、数据库的建立和维护功能等。

**通俗解释**：DBMS就是管理数据库的"管家"——帮你建库、存数据、查数据、删数据，还负责安全和备份。常见的DBMS有MySQL、Oracle、SQL Server、PostgreSQL等。

**详细说明**：DBMS是数据库系统的核心，它介于应用程序和操作系统之间。用户和应用程序通过DBMS来访问数据库，而不是直接操作底层数据文件。DBMS负责数据的物理存储、逻辑组织、安全控制、并发控制、完整性检查、备份恢复等任务。

### 1.1.4 数据库系统（DBS, Database System）

**教材定义**：数据库系统是指在计算机系统中引入数据库后的系统构成，一般由数据库、数据库管理系统（及其开发工具）、应用系统、数据库管理员（DBA）构成。

**通俗解释**：数据库系统 = 数据库 + 管理软件 + 应用程序 + 管理员。它是一个完整的"体系"，不光有数据仓库，还有管理它的工具、使用它的应用，以及维护它的人员。

**详细说明**：DBS是一个广义的概念，它包括了所有的硬件和软件资源以及人员。DB只是其中的一部分（数据的集合），DBMS是管理软件，应用程序是使用数据的前端系统，DBA是负责设计、维护和管理数据库的专业人员。

### 1.1.5 数据库管理员（DBA, Database Administrator）

**教材定义**：数据库管理员是负责数据库的建立、维护和管理的人员。

**通俗解释**：DBA就是"数据库管家"的负责人，负责建库、管库、调优、备份、授权、处理故障等。

**详细说明**：DBA的具体职责包括：(1) 决定数据库中的信息内容和结构；(2) 决定数据库的存储结构和存取策略；(3) 定义数据的安全性要求和完整性约束条件；(4) 监控数据库的使用和运行；(5) 数据库的改进和重组。

### 1.1.6 DBS/DB/DBMS/DBA 区别对比表

| 概念 | 全称 | 含义 | 类比 |
|------|------|------|------|
| DB | Database | 数据库 | 仓库（货物的集合） |
| DBMS | Database Management System | 数据库管理系统 | 仓库管理系统（软件） |
| DBS | Database System | 数据库系统 | 整个仓储体系（仓库+系统+人员+流程） |
| DBA | Database Administrator | 数据库管理员 | 仓库管理员（人） |

**关键区别**：DBS是最大的概念，包含了DB、DBMS、DBA和应用程序。DB是数据的集合，DBMS是管理软件，DBA是人。

## 1.2 数据库系统的三级模式结构

### 1.2.1 三级模式概述

**教材定义**：数据库系统的三级模式结构是指数据库系统由外模式、模式和内模式三级构成。

**通俗解释**：就像一栋大楼——内模式是地基（怎么在硬盘上存数据），模式是标准楼层（数据的逻辑结构），外模式是每个租户看到的装修（用户看到的数据视图）。

**详细说明**：三级模式结构是数据库系统的核心架构，它通过分层来实现数据独立性，使得用户、应用程序和底层存储之间可以解耦。这种设计使得数据库的逻辑结构变化不影响物理存储，物理存储的变化也不影响用户程序。

### 1.2.2 模式（Schema）—— 也称逻辑模式

**教材定义**：模式是数据库中全体数据的逻辑结构和特征的描述，是所有用户的公共数据视图。一个数据库只有一个模式。

**通俗解释**：模式就是数据库的"全局设计图"，定义了所有表的结构、字段名、数据类型、约束等。所有用户看到的数据结构都基于这个全局蓝图。

**详细说明**：模式是数据库系统模式结构的中间层，它既不涉及数据的物理存储细节，也不涉及应用程序、开发工具等。模式是数据库管理员（DBA）和数据库设计者所关心的内容。模式的定义语言称为模式DDL（Schema DDL）。

### 1.2.3 外模式（External Schema）—— 也称子模式或用户模式

**教材定义**：外模式是数据库用户（包括应用程序员和最终用户）能够看见和使用的局部数据的逻辑结构和特征的描述，是数据库用户的数据视图，是与某一应用有关的数据的逻辑表示。一个数据库可以有多个外模式。

**通俗解释**：外模式就是给不同用户定制的"数据窗口"。比如HR看到员工信息表，财务看到薪资表，他们看到的数据只是数据库的一部分。同一个数据库，不同角色看到不同的外模式。

**详细说明**：外模式是模式的子集，一个数据库可以有多个外模式。外模式保证了数据库的安全性——每个用户只能看到和操作与自己相关的数据。外模式的定义语言称为外模式DDL。

### 1.2.4 内模式（Internal Schema）—— 也称存储模式

**教材定义**：内模式是数据物理结构和存储方式的描述，是数据在数据库内部的表示方式。一个数据库只有一个内模式。

**通俗解释**：内模式就是数据在硬盘上的"底层存储方案"——用什么文件格式、什么索引方式、数据怎么排列。这是DBMS最底层的实现细节。

**详细说明**：内模式定义了数据的存储结构和存取方法，如记录的存储方式（顺序存储、哈希存储等）、索引的组织方式、数据是否压缩、是否加密等。内模式通常由DBMS自动管理，用户一般不需要关心。

### 1.2.5 两级映像

**教材定义**：数据库系统在三级模式之间提供了两级映像：外模式/模式映像和模式/内模式映像。

| 映像类型 | 作用 | 对应的数据独立性 |
|---------|------|-----------------|
| 外模式/模式映像 | 保证当模式改变时，外模式保持不变（通过修改映像） | 逻辑数据独立性 |
| 模式/内模式映像 | 保证当内模式改变时，模式保持不变（通过修改映像） | 物理数据独立性 |

**通俗解释**：两级映像就像"翻译器"——外模式/模式映像翻译用户看到的数据和实际数据之间的关系，模式/内模式映像翻译逻辑数据和物理存储之间的关系。有了翻译器，改底层存储不影响上面的用户，改上面的视图不影响底层存储。

**详细说明**：外模式/模式映像定义了外模式与模式之间的对应关系。当数据库的模式发生改变时（如增加字段），DBA只需修改外模式/模式映像，使外模式保持不变，从而保证应用程序不需要修改。这就是逻辑数据独立性。模式/内模式映像定义了数据全局逻辑结构与存储结构之间的对应关系。当数据库的存储结构改变时（如更换存储设备或改变存储方式），DBA只需修改模式/内模式映像，使模式保持不变，从而保证应用程序也不需要修改。这就是物理数据独立性。

### 1.2.6 数据独立性

| 类型 | 含义 | 实现机制 |
|------|------|---------|
| 逻辑数据独立性 | 用户的应用程序与数据库的逻辑结构是相互独立的 | 外模式/模式映像 |
| 物理数据独立性 | 用户的应用程序与存储在磁盘数据库中的数据是相互独立的 | 模式/内模式映像 |

**详细说明**：数据独立性是数据库系统的重要目标之一。物理数据独立性更容易实现，因为改变存储方式通常不需要改变逻辑结构。逻辑数据独立性相对更难实现，因为逻辑结构的改变可能涉及多个外模式和应用程序的调整。

### 【例题1】（2017年下半年）

**题目**：在数据库三级模式中，外模式的个数（）。
A. 只能有一个
B. 与用户个数相同
C. 可以有任意多个
D. 由DBA指定数量

**答案**：C

**解析**：外模式是用户的数据视图，一个数据库可以有多个外模式，不同用户可以有不同的外模式。外模式的个数可以有任意多个，没有数量限制。

### 【例题2】（2019年上半年）

**题目**：数据库的三级模式结构中，描述数据库全局逻辑结构和特征的是（）。
A. 外模式
B. 模式
C. 内模式
D. 存储模式

**答案**：B

**解析**：模式（也称逻辑模式）是数据库中全体数据的逻辑结构和特征的描述，是所有用户的公共数据视图。外模式是局部数据视图，内模式是物理存储描述。

### 【例题3】（2021年上半年）

**题目**：保证数据的物理独立性的是（）。
A. 外模式/模式映像
B. 模式/内模式映像
C. 外模式/内模式映像
D. 模式/外模式映像

**答案**：B

**解析**：模式/内模式映像保证了当数据库的内模式（物理存储结构）发生改变时，由DBA对模式/内模式映像做出相应改变，可以使模式保持不变，从而保证应用程序也不需要修改。这就是物理数据独立性。外模式/模式映像保证的是逻辑数据独立性。

### 【例题4】（2022年下半年）

**题目**：以下关于数据库三级模式结构的说法中，正确的是（）。
A. 一个数据库可以有多个模式
B. 外模式是模式的子集
C. 内模式的个数与表的个数相同
D. 三级模式结构不包含映像

**答案**：B

**解析**：外模式是模式的子集，是对模式的局部抽取。一个数据库只有一个模式和一个内模式，但可以有多个外模式。三级模式结构之间通过两级映像来联系。

### 【学习提示】

- **记忆口诀**："外模内三模式，外模/模、模/内两映像"
- **考试技巧**：记住哪个映像对应哪种独立性——"模式在前面的映像，保证的是后面那层的独立性"。外模式/模式映像 → 模式变不影响外模式 → 逻辑独立性。模式/内模式映像 → 内模式变不影响模式 → 物理独立性。

## 1.3 数据库管理系统（DBMS）的功能

### 1.3.1 DBMS 的主要功能

**教材定义**：

| 功能 | 说明 |
|------|------|
| 数据定义功能 | 提供数据定义语言（DDL），用于定义数据库中的数据对象（表、视图、索引等） |
| 数据操纵功能 | 提供数据操纵语言（DML），实现对数据库的基本操作（查询、插入、修改、删除） |
| 数据库运行管理 | 提供数据安全性、完整性控制，并发控制，数据库恢复等 |
| 数据组织、存储和管理 | 确定以何种文件结构和存取方式在存储及级别上组织数据 |
| 数据库的建立和维护 | 提供数据库初始数据的装入、数据转储、恢复、重组、性能监视等 |

**通俗解释**：DBMS就像一个全能管家——帮你建表（定义）、操作数据（操纵）、保护数据安全（运行管理）、管理硬盘上的数据存放（存储管理）、维护和备份（建立维护）。

**详细说明**：DBMS是数据库系统中最核心的软件。它管理数据的整个生命周期，从创建、使用到维护。除了上述五大功能，DBMS还提供：(1) 数据通信功能——与操作系统的联机处理、分时处理和远程作业输入/输出接口；(2) 数据字典——存放数据库各级模式的描述信息，是DBMS管理数据的依据。

### 1.3.2 DDL 与 DML 对比

| 对比项 | DDL（数据定义语言） | DML（数据操纵语言） |
|--------|-------------------|-------------------|
| 作用 | 定义数据库结构 | 操作数据库中的数据 |
| 关键字 | CREATE、ALTER、DROP | SELECT、INSERT、UPDATE、DELETE |
| 操作对象 | 表、视图、索引、约束 | 表中的数据记录 |
| 影响 | 改变数据库的结构 | 改变数据库中的数据值 |

### 1.3.3 DBMS 的其他功能

- **数据通信功能**：与操作系统的联机处理、分时处理和远程作业输入/输出接口
- **数据字典（DD, Data Dictionary）**：存放数据库各级模式的描述信息，包括数据项的名称、类型、长度、约束等，是DBMS管理数据的依据

### 【例题1】（2018年下半年）

**题目**：以下不属于DBMS功能的是（）。
A. 数据定义功能
B. 数据操纵功能
C. 数据库的建立和维护
D. 数据库的概念设计

**答案**：D

**解析**：数据库的概念设计是数据库设计过程中的一个阶段，由DBA或系统分析员完成，不属于DBMS的功能。DBMS的功能包括数据定义、数据操纵、运行管理、存储管理和建立维护。

### 【例题2】（2020年上半年）

**题目**：使用SQL的（）语句可以创建数据库表。
A. DML
B. DDL
C. DCL
D. DQL

**答案**：B

**解析**：CREATE TABLE 属于DDL（数据定义语言），用于定义数据库中的对象（表、视图等）。DML用于操纵数据（INSERT、UPDATE、DELETE等），DCL用于控制权限（GRANT、REVOKE等）。

### 【例题3】（2023年上半年）

**题目**：数据字典的主要作用是（）。
A. 存储用户数据
B. 存放数据库各级模式的描述信息
C. 管理用户权限
D. 实现数据加密

**答案**：B

**解析**：数据字典（Data Dictionary）存放数据库各级模式的描述信息，包括数据项的名称、类型、长度、约束等，是DBMS管理数据的依据。数据字典是DBMS的重要组成部分，不存储用户数据本身。

### 【学习提示】

- **考试技巧**：区分DDL/DML/DCL是常考知识点。记忆方法：DDL定义结构（Define），DML操作数据（Manipulate），DCL控制权限（Control）。
- **重点**：DBMS的五大功能是选择题常考内容，注意不要把"概念设计"归为DBMS的功能。

## 2.1 数据模型的三要素

### 2.1.1 概述

**教材定义**：数据模型是数据库系统的核心和基础。任何一个数据库管理系统都是基于某种数据模型的。数据模型通常由数据结构、数据操作和完整性约束三个要素组成。

**通俗解释**：数据模型就像建筑设计图——数据结构是"房子有哪些房间（表、字段）"，数据操作是"能在房子里做什么（增删改查）"，完整性约束是"房子里的规矩（不能随便拆墙）"。

### 2.1.2 三要素详解

| 要素 | 含义 | 举例 |
|------|------|------|
| 数据结构 | 描述系统的静态特征，即数据对象本身的类型和内容以及数据对象之间的联系 | 关系模型中的关系（表）、属性（列）、元组（行） |
| 数据操作 | 描述系统的动态特征，即对数据库中各种对象的实例允许执行的操作的集合 | 查询、插入、修改、删除 |
| 完整性约束 | 描述数据模型中数据及其联系所具有的制约和依存规则，用以限定符合数据模型的数据库状态及其变化 | 实体完整性（主码非空）、参照完整性（外码约束）、用户自定义完整性 |

**详细说明**：
- **数据结构**是刻画数据模型性质最重要的方面。数据结构是对系统静态特性的描述，它规定了数据模型的基本结构。不同的数据结构决定了不同的数据模型类型。
- **数据操作**是对数据库中各种对象的实例允许执行的操作的集合，包括操作及有关的操作规则。数据库主要有检索和更新（包括插入、删除、修改）两大类操作。
- **完整性约束**是一组完整性规则的集合，它给定数据模型中数据及其联系所具有的制约和依存规则，用以限定符合数据模型的数据库状态以及状态的变化，以保证数据的正确、有效和相容。

### 【例题1】（2019年下半年）

**题目**：数据模型的三个要素中，规定了数据模型的基本结构的是（）。
A. 数据结构
B. 数据操作
C. 完整性约束
D. 数据定义

**答案**：A

**解析**：数据结构是对系统静态特性的描述，它规定了数据模型的基本结构。数据操作描述动态特征，完整性约束规定数据的规则。

### 【例题2】（2022年上半年）

**题目**：完整性约束属于数据模型三要素之一，以下属于完整性约束的是（）。
A. SELECT 查询
B. 实体完整性（主码非空）
C. 关系模型
D. 数据结构

**答案**：B

**解析**：完整性约束包括实体完整性、参照完整性和用户自定义完整性。实体完整性要求主码不能为空，这就是一种完整性约束。SELECT 查询属于数据操作，关系模型属于数据结构。

## 2.2 概念数据模型（E-R 模型）

### 2.2.1 概述

**教材定义**：概念模型是从用户的角度看到的数据库的模型，它与具体的DBMS无关。最常用的概念模型是E-R模型（实体-联系模型）。

**通俗解释**：E-R模型就是在建数据库之前，先画一张"草图"——哪些东西（实体）、这些东西有什么特征（属性）、它们之间什么关系（联系）。

### 2.2.2 实体（Entity）

**教材定义**：客观存在并可相互区别的事物称为实体。实体可以是具体的人、事、物，也可以是抽象的概念或联系。

**通俗解释**：实体就是一个"东西"——学生、课程、订单都是实体。实体在E-R图中用矩形表示。

**详细说明**：
- **实体**：客观存在并可相互区别的事物，如一个学生、一门课程
- **实体集（Entity Set）**：同一类型实体的集合，如全体学生
- **实体型**：用实体名及其属性名集合来抽象刻画同类实体，如学生(学号, 姓名, 年龄)

### 2.2.3 属性（Attribute）

**教材定义**：实体所具有的某一特性称为属性。一个实体可以由若干个属性来刻画。

**通俗解释**：属性就是实体的"特征"——学生的姓名、年龄、性别等都是属性。属性在E-R图中用椭圆形表示。

**属性的分类**：

| 分类方式 | 类型 | 说明 | 举例 |
|---------|------|------|------|
| 按取值 | 简单属性 | 不可再分的属性 | 年龄 |
| 按取值 | 复合属性 | 可以划分为更小的属性 | 地址（省、市、区） |
| 按唯一性 | 单值属性 | 只有一个值的属性 | 姓名 |
| 按唯一性 | 多值属性 | 可以有多个值的属性 | 电话号码 |
| 按派生 | 基本属性 | 实际存储的属性 | 出生日期 |
| 按派生 | 派生属性 | 由其他属性计算得出 | 年龄（由出生日期计算） |

### 2.2.4 联系（Relationship）

**教材定义**：现实世界中事物内部以及事物之间的联系在信息世界中反映为实体内部的联系和实体之间的联系。

**通俗解释**：联系就是实体之间的"关系"——学生"选修"课程，这就是一个联系。联系在E-R图中用菱形表示。

**联系的类型**：

| 类型 | 符号 | 说明 | 举例 |
|------|------|------|------|
| 一对一（1:1） | 1:1 | A中一个实体至多联系B中一个实体，反之亦然 | 班级与班长（一个班一个班长，一个班长只管一个班） |
| 一对多（1:n） | 1:n | A中一个实体可联系B中多个实体，但B中一个实体至多联系A中一个实体 | 班级与学生（一个班多个学生，一个学生只属于一个班） |
| 多对多（m:n） | m:n | A中一个实体可联系B中多个实体，B中一个实体也可联系A中多个实体 | 学生与课程（一个学生选多门课，一门课有多个学生选） |

### 2.2.5 E-R 图绘制规则

| 元素 | 图形 |
|------|------|
| 实体 | 矩形 □ |
| 属性 | 椭圆形 ○ |
| 联系 | 菱形 ◇ |
| 码（主属性） | 属性名下加下划线 |
| 多值属性 | 椭圆形外加括号（○） |
| 派生属性 | 虚线椭圆 |

### 【例题1】（2018年上半年）

A. 矩形
B. 椭圆形

D. 三角形

**答案**：C

**解析**：E-R图中，矩形表示实体，椭圆形表示属性，菱形表示实体间的联系。没有三角形这种元素。

## 2.3 基本数据模型

### 2.3.1 概述

**教材定义**：基本数据模型是按照计算机系统的观点对数据建模，主要用于DBMS的实现。常用的基本数据模型有：层次模型、网状模型和关系模型。

**通俗解释**：概念模型（E-R模型）是给人看的"草图"，基本数据模型是给计算机看的"正式设计图"。

### 2.3.2 三种基本数据模型对比

| 对比项 | 层次模型 | 网状模型 | 关系模型 |
|--------|---------|---------|---------|
| 数据结构 | 树形结构（有向树） | 图结构（有向图） | 二维表（关系） |
| 联系表示 | 只能表示1:n | 可以表示1:n和m:n（需转换） | 可以表示1:1、1:n、m:n |
| 典型代表 | IBM的IMS | CODASYL的DBTG | Oracle、MySQL、SQL Server |
| 优点 | 结构简单、查询效率高 | 能直接描述复杂联系 | 结构简单、概念单一、语言一体化 |
| 缺点 | 不能直接表示m:n联系 | 结构复杂、DDL和DML复杂 | 查询效率不如层次和网状 |
| 当前使用 | 基本淘汰 | 基本淘汰 | 当前主流 |

**详细说明**：
- **层次模型**：用树形结构来表示各类实体以及实体间的联系。每个节点表示一个记录类型，记录之间的联系用连线表示。优点是结构简单清晰，查询效率高。缺点是不能直接表示多对多的联系，查询子女节点必须通过双亲节点。
- **网状模型**：用图结构（有向图）来表示各类实体以及实体间的联系。允许一个以上的节点无双亲，一个节点可以有多于一个的双亲。优点是能更直接地描述现实世界。缺点是结构复杂，语言复杂。
- **关系模型**：用二维表（关系）来表示数据和数据之间的联系。是目前最流行的数据模型。关系模型建立在严格的数学概念基础上（集合代数）。关系模型中的数据操作是集合操作，操作对象和操作结果都是关系（元组的集合）。

### 【例题1】（2017年上半年）

**题目**：以下数据模型中，用二维表来表示数据的是（）。
A. 层次模型
B. 网状模型
C. 关系模型
D. 面向对象模型

**答案**：C

**解析**：关系模型用二维表（关系）来表示数据和数据之间的联系。层次模型用树形结构，网状模型用图结构。

### 【例题2】（2020年下半年）

**题目**：层次模型和网状模型的共同缺点是（）。
A. 不能表示多对多联系
B. 结构过于简单
C. 数据独立性差
D. 查询语言复杂

**答案**：C

**解析**：层次模型和网状模型的一个共同缺点是数据独立性较差，应用程序在存取数据时必须清楚数据库的逻辑结构和物理结构。层次模型不能直接表示m:n联系，网状模型结构复杂且语言复杂，这些是各自不同的缺点。

## 2.4 E-R 图向关系模型的转换（★★★ 下午题必考）

### 2.4.1 转换规则

**教材定义**：E-R图向关系模型的转换要解决的问题是如何将实体型和实体间的联系转换为关系模式。

**通俗解释**：把E-R图（草图）变成数据库表（正式设计），需要一套规则来处理不同类型的联系。

### 2.4.2 实体的转换

**规则**：一个实体型转换为一个关系模式。实体的属性就是关系的属性，实体的码就是关系的码。

**示例**：
- 实体"学生"（学号, 姓名, 年龄）→ 关系：学生(学号, 姓名, 年龄)，学号为主码
- 实体"课程"（课程号, 课程名, 学分）→ 关系：课程(课程号, 课程名, 学分)，课程号为主码

### 2.4.3 联系的转换（★★★ 核心考点）

| 联系类型 | 转换规则 | 示例 |
|---------|---------|------|
| **1:1** | 方案一：将联系合并到任一方实体的关系模式中（在该关系中加入对方的主码和联系的属性） | 班级(班号, 班名, 班长学号) — 将1:1联系合并到班级表 |
| **1:1** | 方案二：将联系单独转换为一个关系模式（两个实体的主码+联系属性） | 管理(班号, 学号, 任职时间) |
| **1:n** | 方案一：将联系合并到n端实体的关系模式中（在n端加入1端的主码和联系属性） | 学生(学号, 姓名, 班号) — 将1:n联系合并到n端 |
| **1:n** | 方案二：将联系单独转换为一个关系模式 | 归属(班号, 学号) |
| **m:n** | **必须**将联系单独转换为一个关系模式（两个实体的主码+联系属性） | 选修(学号, 课程号, 成绩) — m:n联系必须单独建表 |

### 2.4.4 关键要点

1. **1:1和1:n联系**：可以合并到实体的关系模式中，也可以单独建表
2. **m:n联系**：**必须**单独转换为一个关系模式，不能合并
3. 转换时，联系的属性也要一并处理
4. 同一实体内部的联系也要按照上述规则处理

### 【例题1】（2018年下半年 下午题）

**题目**：某学校管理系统有如下E-R图：
- 实体：学生（学号, 姓名, 性别）
- 实体：课程（课程号, 课程名, 学分）
- 联系：选修（成绩），m:n联系

请将上述E-R图转换为关系模式，并指出每个关系的主码和外码。

**答案**：
- 学生(学号, 姓名, 性别)，主码：学号
- 课程(课程号, 课程名, 学分)，主码：课程号
- 选修(学号, 课程号, 成绩)，主码：(学号, 课程号)，外码：学号参照学生(学号)，课程号参照课程(课程号)

**解析**：两个实体和一个m:n联系，m:n联系必须单独转换为关系模式。选修表的主码是两个实体主码的组合，外码分别参照对应的实体表。

### 【例题2】（2019年下半年 下午题）

**题目**：设有部门和员工两个实体，部门（部门号, 部门名），员工（工号, 姓名），一个部门有多名员工，一名员工只属于一个部门，联系中有"入职日期"属性。请转换为关系模式。

**答案**：
- 方案一（推荐）：员工(工号, 姓名, 部门号, 入职日期)，主码：工号，外码：部门号
  部门(部门号, 部门名)，主码：部门号
- 方案二：员工(工号, 姓名)，部门(部门号, 部门名)，归属(工号, 部门号, 入职日期)

**解析**：部门和员工是1:n联系，联系属性"入职日期"属于联系。方案一是将联系合并到n端（员工表），在员工表中加入1端的主码（部门号）和联系属性（入职日期）。方案二是将联系单独建表。

### 【例题3】（2021年上半年 下午题）

**题目**：设有实体A（a1, a2）和实体B（b1, b2），A与B之间是1:1联系，联系的属性为c。请给出转换为关系模式的两种方案。

**答案**：
- 方案一：将联系合并到A（或B）：
  - A(a1, a2, b1, c)，主码：a1，外码：b1
  - B(b1, b2)，主码：b1
- 或合并到B：
  - A(a1, a2)，主码：a1
  - B(b1, b2, a1, c)，主码：b1，外码：a1
- 方案二：将联系单独建表：
  - A(a1, a2)，主码：a1
  - B(b1, b2)，主码：b1
  - AB(a1, b1, c)，主码：a1 或 b1（任选），外码分别参照A和B

**解析**：1:1联系可以合并到任一方，也可以单独建表。合并时在选择的一方加入对方的主码和联系属性。如果单独建表，任一方的主码都可以作为联系表的主码。

### 【例题4】（2022年下半年）

**题目**：一个实体"课程"有属性课程号、课程名、学分，该实体与自身存在"先修"联系（一门课程有多门先修课程），先修联系没有属性。请给出关系模式。

**答案**：
- 课程(课程号, 课程名, 学分)，主码：课程号
- 先修(课程号, 先修课程号)，主码：(课程号, 先修课程号)，两个属性均为外码，参照课程(课程号)

**解析**：实体内部的联系也按照1:1、1:n、m:n的规则处理。这里"一门课程有多门先修课程，一门先修课程可以是多门课程的先修"是m:n联系（同一个实体内部），所以必须单独建表。

## 3.1 关系模型的基本概念

### 3.1.1 关系（Relation）

**教材定义**：一个关系对应通常说的一张表。关系就是一张规范化的二维表。

**通俗解释**：关系就是Excel里的一张工作表——有行有列，整整齐齐。

### 3.1.2 基本术语详解

| 术语 | 英文 | 含义 | 对应表格 |
|------|------|------|---------|
| 关系 | Relation | 一张二维表 | 整张表 |
| 元组 | Tuple | 表中的一行 | 一条记录 |
| 属性 | Attribute | 表中的一列 | 一个字段 |
| 域 | Domain | 属性的取值范围 | 如性别的域为{男, 女} |
| 分量 | Component | 元组中的一个属性值 | 一个单元格的值 |
| 关系模式 | Relation Schema | 对关系的描述（关系名+属性名集合） | 学生(学号, 姓名, 年龄) |
| 候选码 | Candidate Key | 能唯一标识元组的属性或属性组（最小） | 如学号、身份证号 |
| 主码/主键 | Primary Key | 从候选码中选定的一个 | 学号（选定） |
| 主属性 | Prime Attribute | 包含在任一候选码中的属性 | |
| 非主属性 | Non-prime Attribute | 不包含在任何候选码中的属性 | |
| 外码/外键 | Foreign Key | 关系R中的属性是另一关系S的主码 | 学生表的"班号"是班级表的主码 |
| 全码 | All-key | 关系的所有属性构成候选码 | |

**详细说明**：
- **候选码**：能唯一标识元组的属性或属性组，且该属性组的任何真子集都不能唯一标识元组（最小性）。一个关系可以有多个候选码。
- **主码**：从候选码中选定的一个，用于唯一标识元组。主码不能为空，不能重复。
- **外码**：如果关系R的某个属性（组）F不是R的主码，但它是另一个关系S的主码，则F是R的外码。外码实现了关系之间的联系。

### 【例题1】（2018年下半年）

**题目**：在关系模型中，以下说法正确的是（）。
A. 关系中元组的顺序是重要的
B. 关系中属性的顺序是重要的
C. 关系中不允许有重复的元组
D. 关系中不允许有空值

**答案**：C

**解析**：关系模型中，元组的顺序和属性的顺序都不重要（可以交换），这是关系的两个重要性质。关系中不允许有重复的元组（元组唯一性）。主属性不允许为空值，但非主属性在特定条件下可以为空。

## 3.2 关系代数运算

### 3.2.1 概述

**教材定义**：关系代数是一种抽象的查询语言，是关系数据操纵语言的一种传统表达方式。关系代数的运算对象是关系，运算结果也是关系。

**通俗解释**：关系代数就是对表进行"数学运算"——合并表、筛选行、选列、连接等。

### 3.2.2 传统的集合运算

#### （1）并运算（∪）

**定义**：R∪S = {t | t∈R ∨ t∈S}，即R和S的并集，结果是属于R或属于S的所有元组。

**要求**：R和S必须具有相同的关系模式（相同的属性个数、属性名、属性顺序）。

**示例**：
- R: {1,2,3}, S: {2,3,4} → R∪S = {1,2,3,4}

#### （2）差运算（−）

**定义**：R−S = {t | t∈R ∧ t∉S}，即属于R但不属于S的所有元组。

**示例**：
- R: {1,2,3}, S: {2,3,4} → R−S = {1}

#### （3）笛卡尔积（×）

**定义**：R×S = {tr ts | tr∈R ∧ ts∈S}，将R中的每个元组与S中的每个元组拼接。

**属性**：结果的属性数 = R的属性数 + S的属性数
**元组数**：结果的元组数 = R的元组数 × S的元组数

**示例**：R有m个属性、p个元组，S有n个属性、q个元组 → R×S有(m+n)个属性、(p×q)个元组。

### 3.2.3 专门的关系运算

#### （1）选择运算（σ）—— 行运算

**定义**：σ_F(R) = {t | t∈R ∧ F(t)=true}，从关系R中选取满足条件F的元组。

**通俗解释**：选择就是"按条件筛行"——选出满足条件的行。

**示例**：σ_{年龄>20}(学生) → 选出学生表中年龄大于20的所有行。

#### （2）投影运算（π）—— 列运算

**定义**：π_A(R) = {t[A] | t∈R}，从关系R中选取若干属性列组成新的关系。

**通俗解释**：投影就是"选列"——只看需要的列，去掉不需要的列。

**注意**：投影后可能会消除重复的元组（因为属性少了，可能有重复行）。

**示例**：π_{姓名, 年龄}(学生) → 只取学生表的姓名和年龄两列。

### 3.2.4 五种基本运算总结

| 运算 | 符号 | 类型 | 说明 |
|------|------|------|------|
| 并 | ∪ | 集合 | R和S的所有元组（去重） |
| 差 | − | 集合 | 属于R但不属于S的元组 |
| 笛卡尔积 | × | 集合 | R的每个元组与S的每个元组组合 |
| 选择 | σ | 关系 | 按条件选取行 |
| 投影 | π | 关系 | 选取若干列 |

**关键记忆**：关系代数的5种基本运算是并(∪)、差(−)、笛卡尔积(×)、选择(σ)、投影(π)。其他运算都可以用这5种来表示。

### 【例题1】（2019年上半年）

**题目**：关系代数中，选择运算是从关系中找出满足给定条件的元组，它是一种（）运算。
A. 垂直方向（列）
B. 水平方向（行）
C. 水平和垂直方向
D. 连接

**答案**：B

**解析**：选择运算是按条件选取行，是水平方向的运算（"水平切"）。投影运算是选取列，是垂直方向的运算（"垂直切"）。

### 【例题2】（2020年上半年）

**题目**：设有关系R(A,B,C)，其中A、B、C是属性名。则π_{A,C}(σ_{B='b1'}(R)) 的含义是（）。
A. 从R中选择B='b1'的元组，然后投影出A和C两列
B. 从R中投影出A和C两列，然后选择B='b1'的元组
C. 从R中选择A='b1'的元组，然后投影出B和C两列
D. 从R中选择C='b1'的元组，然后投影出A和B两列

**答案**：A

**解析**：先做选择σ_{B='b1'}(R)，从R中选出B='b1'的行。再做投影π_{A,C}，选出A和C两列。关系代数表达式从内向外求值。

### 【例题3】（2021年下半年）

**题目**：设有关系R有3个属性、5个元组，关系S有2个属性、4个元组，则R×S有（）。
A. 5个属性，9个元组
B. 5个属性，20个元组
C. 6个属性，9个元组
D. 6个属性，20个元组

**答案**：B

**解析**：笛卡尔积的结果属性数 = R属性数 + S属性数 = 3 + 2 = 5。结果元组数 = R元组数 × S元组数 = 5 × 4 = 20。

## 3.3 扩展关系代数运算

### 3.3.1 交运算（∩）

**定义**：R∩S = {t | t∈R ∧ t∈S}，属于R且属于S的所有元组。

**与基本运算的关系**：R∩S = R − (R−S)

### 3.3.2 连接运算（⋈）

**定义**：R ⋈_θ S = σ_{R.A θ S.B}(R×S)，其中θ是比较运算符。

**通俗解释**：先做笛卡尔积，再按条件选行。就是"有条件的笛卡尔积"。

**等值连接**：θ为"="时的连接，即R ⋈_{A=B} S。

### 3.3.3 自然连接（⋈）

**定义**：R ⋈ S，先进行R和S的笛卡尔积，然后选取公共属性上值相等的元组，最后去掉重复的公共属性列。

**通俗解释**：自然连接 = 等值连接 + 去重复列。它自动找公共属性名，按公共属性值相等进行连接。

**重要公式**：如果R有m个属性、p个元组，S有n个属性、q个元组，且R和S有k个公共属性，则：
- 自然连接结果的属性数 = m + n − k
- 自然连接结果的元组数 ≤ p × q（取决于匹配情况）

**等值连接 vs 自然连接对比**：

| 对比项 | 等值连接 | 自然连接 |
|--------|---------|---------|
| 公共属性 | 可以不同名 | 必须同名 |
| 重复列 | 保留 | 去除 |
| 连接条件 | 需要手动指定 | 自动按公共属性 |

### 3.3.4 除运算（÷）

**定义**：设关系R(X,Y)和S(Y,Z)，则 R÷S = {t[X] | t∈R ∧ π_Y(S) ⊆ Y_x}
其中Y_x是x在R中的Y属性组的值集合。

**通俗解释**：除法找的是"对S中所有Y值都有对应记录的X值"。简单说：R中某个X值，如果它对应的Y值包含了S中所有Y值，则该X值属于结果。

**示例**：R(学号, 课程号)，S(课程号) = {C1, C2}。R÷S找出选修了C1和C2两门课程的学生学号。

### 3.3.5 扩展运算总结

| 运算 | 符号 | 由基本运算表示 |
|------|------|---------------|
| 交 | ∩ | R∩S = R − (R−S) |
| 连接 | ⋈_θ | R⋈_θS = σ_θ(R×S) |
| 自然连接 | ⋈ | 等值连接后去重复列 |
| 除 | ÷ | 需用π、σ、−、×组合表示 |

### 【例题1】（2019年下半年）

**题目**：关系R(A,B)的值如下：{(a,1), (b,2), (c,3)}，关系S(B,C)的值如下：{(1,x), (2,y)}。则R⋈S的结果是（）。

**答案**：{(a,1,x), (b,2,y)}

**解析**：自然连接按公共属性B进行等值连接，然后去掉重复的公共属性列B（保留一个）。
- (a,1)与(1,x)：B值相等→(a,1,x)
- (b,2)与(2,y)：B值相等→(b,2,y)
- (c,3)在S中找不到B=3的元组，不参与连接

### 【例题2】（2020年下半年）

**题目**：设有关系R(学号, 课程号, 成绩)和S(课程号)。如果S={(C1),(C2)}，R中选修了C1和C2两门课的学生学号可以用（）运算求出。
A. R÷S
B. R−S
C. R∩S
D. R⋈S

**答案**：A

**解析**：除运算用于"对S中所有值都有对应记录"的查询。R÷S的结果是选修了S中所有课程的学生学号。这是除运算的典型应用场景。

### 【例题3】（2022年上半年）

**题目**：关系代数中，（）运算是从行的角度进行的，（）运算是从列的角度进行的。
A. 选择、投影
B. 投影、选择
C. 连接、选择
D. 选择、连接

**答案**：A

**解析**：选择运算是按条件选取满足条件的行（水平方向），投影运算是选取若干属性列（垂直方向）。

## 3.4 关系代数综合计算题

### 3.4.1 常见题型与解题思路

**题型一：写出关系代数表达式**

**例题**（2021年上半年）：设有学生表S(学号, 姓名, 年龄, 性别)，课程表C(课程号, 课程名, 学分)，选课表SC(学号, 课程号, 成绩)。

**问**：查询选修了"数据库"课程的学生姓名。

**解析**：
1. 先从课程表中找出课程名为"数据库"的课程号：σ_{课程名='数据库'}(C)
2. 从选课表中找出选了该课程号的学号：π_{学号}(SC ⋈ σ_{课程名='数据库'}(C))
3. 从学生表中找出对应学生的姓名：π_{姓名}(S ⋈ π_{学号}(SC ⋈ σ_{课程名='数据库'}(C)))

**答案**：π_{姓名}(S ⋈ π_{学号}(SC ⋈ σ_{课程名='数据库'}(C)))

### 3.4.2 答题模板

1. **分析题目**：找出涉及哪些表、哪些条件、要输出什么
2. **分步构造**：
   - 第一步：选择（σ）—— 按条件筛选
   - 第二步：连接（⋈）—— 关联多表
   - 第三步：投影（π）—— 输出需要的列
3. **由内到外**：先写内部的运算，再嵌套外部的运算

### 【例题】（2022年下半年）

**题目**：设有关系R和S如下：
- R(A,B): {(1,2), (3,4), (5,6)}
- S(B,C): {(2,7), (4,8), (6,9)}

计算：(1) R÷π_B(S)  (2) σ_{A>2}(R) ⋈ S

**答案**：
(1) π_B(S) = {(2), (4), (6)}。R中每个A值对应的B值集合分别是：A=1→{2}、A=3→{4}、A=5→{6}。没有哪个A值的B集合包含π_B(S)的全部值，所以R÷π_B(S) = ∅（空集）。

(2) σ_{A>2}(R) = {(3,4), (5,6)}
(σ_{A>2}(R)) ⋈ S = {(3,4,8), (5,6,9)}

### 【学习提示】

- **考试技巧**：选择（σ）是"行操作"，投影（π）是"列操作"，这是最常考的区分
- **自然连接**：注意公共属性的处理——按公共属性等值连接后去重复列
- **除运算**：考试频率不高，但理解"包含所有"的概念很重要
- **高频考点**：给关系代数表达式，求结果；或给查询要求，写关系代数表达式

## 4.1 SQL 概述

### 4.1.1 SQL 简介

**教材定义**：SQL（Structured Query Language，结构化查询语言）是关系数据库的标准语言，是一种通用的、功能强大的关系数据库语言。

**通俗解释**：SQL就是跟数据库"说话"的语言——告诉它我要查什么、改什么、建什么。

**详细说明**：SQL最早由IBM在20世纪70年代开发，后来被ANSI和ISO采纳为关系数据库标准语言。SQL具有以下特点：(1) 综合统一——集DDL、DML、DCL于一体；(2) 高度非过程化——只需说明"做什么"，不需要说明"怎么做"；(3) 面向集合的操作方式——操作对象和结果都是集合；(4) 同一种语法结构提供两种使用方式——既可独立使用（交互式），也可嵌入到高级语言中使用（嵌入式）；(5) 语言简洁、易学易用。

### 4.1.2 SQL 的语言分类

| 分类 | 全称 | 功能 | 关键字 |
|------|------|------|--------|
| DDL | Data Definition Language | 数据定义（定义表、视图、索引等） | CREATE、ALTER、DROP |
| DML | Data Manipulation Language | 数据操纵（增删改查） | SELECT、INSERT、UPDATE、DELETE |
| DCL | Data Control Language | 数据控制（权限管理） | GRANT、REVOKE |
| TCL | Transaction Control Language | 事务控制 | COMMIT、ROLLBACK、SAVEPOINT |

**通俗解释**：
- DDL："画表格"——建表、改表、删表
- DML："填表格"——查数据、加数据、改数据、删数据
- DCL："给钥匙"——谁能看什么、谁能改什么
- TCL："确认/撤回"——提交事务、回滚事务

### 【例题1】（2019年下半年）

**题目**：SQL语言中，用于实现数据查询的语句是（）。
A. SELECT
B. INSERT
C. UPDATE
D. DELETE

**答案**：A

**解析**：SELECT用于查询数据，INSERT用于插入数据，UPDATE用于修改数据，DELETE用于删除数据。

### 【例题2】（2021年上半年）

**题目**：以下SQL语句中，属于DDL的是（）。
A. SELECT
B. INSERT
C. CREATE TABLE
D. GRANT

**答案**：C

**解析**：CREATE TABLE 属于DDL（数据定义语言）。SELECT和INSERT属于DML，GRANT属于DCL。

## 4.2 数据定义

### 4.2.1 CREATE TABLE

**语法**：
```sql
CREATE TABLE 表名 (
    列名1 数据类型 [列级完整性约束],
    列名2 数据类型 [列级完整性约束],
    ...
    [表级完整性约束]
);
```

**示例**：
```sql
CREATE TABLE Student (
    Sno CHAR(9) PRIMARY KEY,       -- 学号，主码
    Sname VARCHAR(20) NOT NULL,    -- 姓名，非空
    Sage INT CHECK(Sage>=1 AND Sage<=150),  -- 年龄，检查约束
    Ssex CHAR(2) DEFAULT '男',     -- 性别，默认值
    Sdept VARCHAR(20)              -- 系别
);
```

**常见数据类型**：

| 数据类型 | 说明 | 举例 |
|---------|------|------|
| CHAR(n) | 固定长度字符串 | CHAR(9) 存9位学号 |
| VARCHAR(n) | 可变长度字符串 | VARCHAR(20) 最长20位 |
| INT / INTEGER | 整数 | |
| FLOAT / REAL | 浮点数 | |
| DATE | 日期 | '2023-09-01' |
| DECIMAL(p,s) | 定点数 | DECIMAL(5,2) 总5位，小数2位 |

### 4.2.2 ALTER TABLE

**语法**：
```sql
-- 添加列
ALTER TABLE 表名 ADD 列名 数据类型;

-- 修改列的数据类型
ALTER TABLE 表名 ALTER COLUMN 列名 新数据类型;

-- 删除列
ALTER TABLE 表名 DROP COLUMN 列名;

-- 添加约束
ALTER TABLE 表名 ADD CONSTRAINT 约束名 约束定义;

-- 删除约束
ALTER TABLE 表名 DROP CONSTRAINT 约束名;
```

### 4.2.3 DROP TABLE

**语法**：
```sql
DROP TABLE 表名;
```

**说明**：删除表结构和所有数据。有些DBMS支持 `DROP TABLE 表名 CASCADE;`（级联删除相关对象）。

### 4.2.4 CREATE INDEX

**语法**：
```sql
CREATE [UNIQUE] [CLUSTER] INDEX 索引名
ON 表名(列名 [ASC|DESC], ...);
```

**示例**：
```sql
CREATE UNIQUE INDEX Stusno ON Student(Sno);  -- 在Student表的Sno列上建唯一索引
CREATE INDEX SCgrade ON SC(Grade DESC);        -- 在SC表的Grade列上建索引，降序
```

**说明**：
- UNIQUE：唯一索引，索引列的值不能重复
- CLUSTER：聚簇索引，数据按索引顺序物理存储
- 一个表只能有一个聚簇索引，可以有多个非聚簇索引

### 【例题1】（2020年上半年）

**题目**：在SQL中，用CREATE TABLE创建表时，PRIMARY KEY约束表示（）。

**解析**：DROP TABLE用于删除表。DELETE用于删除表中的数据（行），不是删除表本身。

## 4.3 数据查询（SELECT）

### 4.3.1 SELECT 基本语法

```sql
SELECT [ALL|DISTINCT] 目标列
FROM 表名|视图名
[WHERE 条件表达式]
[GROUP BY 列名 [HAVING 条件表达式]]
[ORDER BY 列名 [ASC|DESC]];
```

**执行顺序**：FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY

**通俗解释**：SQL查询就像问问题——"从哪张表里（FROM），找什么样的数据（WHERE），怎么分组统计（GROUP BY），分组后筛什么（HAVING），输出哪些列（SELECT），按什么排序（ORDER BY）"。

### 4.3.2 WHERE 子句

**常用条件**：

| 运算符 | 含义 | 示例 |
|--------|------|------|
| = | 等于 | WHERE Sage = 20 |
| != 或 <> | 不等于 | WHERE Ssex <> '男' |
| >, <, >=, <= | 比较 | WHERE Sage >= 18 |
| AND | 与 | WHERE Sage > 18 AND Ssex = '男' |
| OR | 或 | WHERE Sdept = 'CS' OR Sdept = 'MA' |
| NOT | 非 | WHERE NOT (Sage > 20) |
| BETWEEN...AND... | 范围（包含边界） | WHERE Sage BETWEEN 18 AND 22 |
| IN | 在集合中 | WHERE Sdept IN ('CS', 'MA', 'IS') |
| LIKE | 模式匹配 | WHERE Sname LIKE '张%' |
| IS NULL | 是否为空 | WHERE Sage IS NULL |
| IS NOT NULL | 是否不为空 | WHERE Sage IS NOT NULL |

**LIKE 通配符**：

| 通配符 | 含义 | 示例 |
|--------|------|------|
| % | 任意长度的字符串（包括0） | '张%' 匹配"张三"、"张三丰" |
| _ | 任意单个字符 | '张_' 只匹配两个字的姓张的 |

### 4.3.3 聚合函数

| 函数 | 功能 | 示例 |
|------|------|------|
| COUNT(*) | 统计行数 | SELECT COUNT(*) FROM Student |
| COUNT(列名) | 统计非空行数 | SELECT COUNT(Sage) FROM Student |
| SUM(列名) | 求和 | SELECT SUM(Grade) FROM SC |
| AVG(列名) | 求平均值 | SELECT AVG(Grade) FROM SC |
| MAX(列名) | 求最大值 | SELECT MAX(Grade) FROM SC |
| MIN(列名) | 求最小值 | SELECT MIN(Grade) FROM SC |

**注意**：聚合函数不能直接用在WHERE子句中！要用HAVING子句。

### 4.3.4 GROUP BY 与 HAVING

```sql
-- 按学号分组，求每个学生的平均成绩
SELECT Sno, AVG(Grade) AS AvgGrade
FROM SC
GROUP BY Sno;

-- 按学号分组，找出平均成绩大于80的学生
SELECT Sno, AVG(Grade) AS AvgGrade
FROM SC
GROUP BY Sno
HAVING AVG(Grade) > 80;
```

**WHERE与HAVING的区别**：

| 对比项 | WHERE | HAVING |
|--------|-------|--------|
| 执行时机 | 分组前过滤 | 分组后过滤 |
| 能否用聚合函数 | 不能 | 能 |
| 作用对象 | 行（元组） | 组 |

### 4.3.5 连接查询

```sql
-- 等值连接
SELECT Student.Sno, Sname, Grade
FROM Student, SC
WHERE Student.Sno = SC.Sno;

-- 自然连接（SQL中通过等值连接实现）
SELECT Student.Sno, Sname, Cno, Grade
FROM Student, SC
WHERE Student.Sno = SC.Sno;

-- 多表连接
SELECT Sname, Cname, Grade
FROM Student, SC, Course
WHERE Student.Sno = SC.Sno AND SC.Cno = Course.Cno;

-- 自身连接（给表起别名）
SELECT FIRST.Cno, SECOND.Cpno
FROM Course FIRST, Course SECOND
WHERE FIRST.Cpno = SECOND.Cno;

-- 外连接
SELECT Student.Sno, Sname, Cno, Grade
FROM Student LEFT OUTER JOIN SC ON Student.Sno = SC.Sno;
```

### 4.3.6 子查询

```sql
-- 单行子查询（用 =, >, < 等）
SELECT Sname FROM Student
WHERE Sno = (SELECT Sno FROM SC WHERE Cno = '1' AND Grade = (SELECT MAX(Grade) FROM SC WHERE Cno = '1'));

-- 多行子查询（用 IN, ANY, ALL）
SELECT Sname FROM Student
WHERE Sno IN (SELECT Sno FROM SC WHERE Cno = '1');

-- ANY的用法：>ANY 表示大于子查询结果中的某个值
-- ALL的用法：>ALL 表示大于子查询结果中的所有值
```

### 4.3.7 EXISTS 与 IN

```sql
-- IN方式：查询选修了课程号为'1'的学生姓名
SELECT Sname FROM Student
WHERE Sno IN (SELECT Sno FROM SC WHERE Cno = '1');

-- EXISTS方式：同样的查询
SELECT Sname FROM Student
WHERE EXISTS (SELECT * FROM SC WHERE SC.Sno = Student.Sno AND Cno = '1');

-- NOT EXISTS：查询没有选修课程号为'1'的学生
SELECT Sname FROM Student
WHERE NOT EXISTS (SELECT * FROM SC WHERE SC.Sno = Student.Sno AND Cno = '1');
```

**IN与EXISTS的区别**：

| 对比项 | IN | EXISTS |
|--------|-----|--------|
| 执行方式 | 先执行子查询，将结果用于外查询 | 对外查询的每一行，执行一次子查询 |
| 效率 | 子查询结果集较小时效率高 | 子查询结果集较大时效率高 |
| 使用场景 | 固定值集合的匹配 | 相关子查询 |

## 4.5 视图

### 4.5.1 视图的定义

**教材定义**：视图是从一个或几个基本表（或视图）导出的表。数据库中只存放视图的定义而不存放视图对应的数据，数据仍存放在原来的基本表中。

**通俗解释**：视图就像一个"虚拟窗口"——不实际存储数据，只是保存了一个查询定义。每次用到视图时，就执行一次定义中的查询。

### 4.5.2 视图的操作

```sql
-- 创建视图
CREATE VIEW CS_Student AS
SELECT Sno, Sname, Sage
FROM Student
WHERE Sdept = 'CS';

-- 创建带聚合的视图
CREATE VIEW S_G(Sno, Gavg) AS
SELECT Sno, AVG(Grade)
FROM SC
GROUP BY Sno;

-- 删除视图
DROP VIEW CS_Student;
```

### 4.5.3 视图与表的区别

| 对比项 | 视图（View） | 基本表（Table） |
|--------|-------------|----------------|
| 数据存储 | 不存储数据（虚拟表） | 存储实际数据 |
| 定义方式 | 由查询定义 | 由CREATE TABLE定义 |
| 数据更新 | 受限（部分视图可更新） | 可自由更新 |
| 修改影响 | 修改基本表会影响视图 | 修改视图不影响基本表 |
| 用途 | 安全性、简化查询、逻辑独立性 | 存储实际数据 |

### 4.5.4 视图的作用

1. **简化用户操作**：将复杂的查询定义为视图，用户只需查询视图
2. **安全性**：用户只能看到视图中定义的数据，看不到基本表中的敏感数据
3. **逻辑数据独立性**：视图提供了一定程度的逻辑独立性

### 【例题1】（2019年上半年）

**题目**：关于视图，以下说法正确的是（）。
A. 视图中存储了实际的数据
B. 对视图的操作最终会转化为对基本表的操作
C. 任何视图都可以进行更新操作
D. 视图不能基于多个基本表创建

**答案**：B

**解析**：视图是虚拟表，不存储实际数据（A错）。对视图的操作最终转化为对基本表的操作（B对）。不是所有视图都可以更新，含聚合函数、DISTINCT、GROUP BY等的视图不可更新（C错）。视图可以基于多个基本表创建（D错）。

## 4.6 完整性约束

### 4.6.1 实体完整性（Entity Integrity）

**教材定义**：若属性A是基本关系R的主属性，则A不能取空值。

**通俗解释**：主码（主键）不能为空！主码用来唯一标识一条记录，如果主码为空，就无法标识了。

**SQL实现**：
```sql
CREATE TABLE Student (
    Sno CHAR(9) PRIMARY KEY,  -- 主码约束
    ...
);
```

### 4.6.2 参照完整性（Referential Integrity）

**教材定义**：若属性F是基本关系R的外码，它与基本关系S的主码Ks相对应，则R中每个元组在F上的值必须是：或者取空值，或者等于S中某个元组的主码值。

**通俗解释**：外码要么为空，要么必须在被参照表的主码中存在。不能凭空编一个不存在的参照值。

**SQL实现**：
```sql
CREATE TABLE SC (
    Sno CHAR(9),
    Cno CHAR(4),
    Grade INT,
    PRIMARY KEY (Sno, Cno),
    FOREIGN KEY (Sno) REFERENCES Student(Sno),  -- 参照完整性
    FOREIGN KEY (Cno) REFERENCES Course(Cno)
);
```

**外码约束的处理策略**：
- **拒绝（RESTRICT/NO ACTION）**：违反约束的操作被拒绝（默认）
- **级联（CASCADE）**：删除/更新被参照的元组时，级联删除/更新参照表中相应的元组
- **置空（SET NULL）**：删除/更新被参照的元组时，将参照表中相应的外码值置为空

### 4.6.3 用户自定义完整性（User-defined Integrity）

**教材定义**：针对某一具体应用的数据必须满足的语义要求。

**通俗解释**：根据业务需求自定义的约束规则。比如：年龄必须大于0，性别只能是'男'或'女'。

**SQL实现**：
```sql
CREATE TABLE Student (
    Sno CHAR(9) PRIMARY KEY,
    Sname VARCHAR(20) NOT NULL,        -- 非空约束
    Sage INT CHECK(Sage>=0 AND Sage<=150),  -- 检查约束
    Ssex CHAR(2) CHECK(Ssex IN ('男', '女')),  -- 检查约束
    Ssex CHAR(2) DEFAULT '男'           -- 默认值约束
);
```

### 4.6.4 三种完整性对比

| 完整性类型 | 作用 | 举例 | SQL关键字 |
|-----------|------|------|----------|
| 实体完整性 | 主码唯一且非空 | 学号不能为空 | PRIMARY KEY |
| 参照完整性 | 外码引用有效 | 课程号必须存在 | FOREIGN KEY...REFERENCES |
| 用户自定义完整性 | 业务规则约束 | 年龄>0 | CHECK、NOT NULL、DEFAULT、UNIQUE |

### 【例题1】（2018年上半年）

**题目**：在关系模型中，要求主码不能取空值，这是（）约束。

**解析**：实体完整性规则规定：若属性A是基本关系R的主属性，则A不能取空值。主码不能为空就是实体完整性的要求。

## 5.1 函数依赖

### 5.1.1 概述

**教材定义**：设R(U)是属性集U上的关系模式，X和Y是U的子集。若对于R(U)的任意一个可能的关系r，r中不可能存在两个元组在X上的属性值相等，而在Y上的属性值不等，则称"X函数确定Y"或"Y函数依赖于X"，记作X→Y。

**通俗解释**：如果知道了X的值，就一定能确定Y的值，就说X决定Y（X→Y）。比如学号→姓名，知道了学号就一定能知道姓名。

**详细说明**：函数依赖是属性之间的一种约束关系，是关系模式规范化理论的基础。函数依赖分为三种类型。

### 5.1.2 三种函数依赖

**（1）完全函数依赖**

**定义**：在R(U)中，如果X→Y，并且对于X的任何一个真子集X'，都有X'→Y不成立，则称Y对X完全函数依赖，记作 X ——F→ Y。

**通俗解释**：Y的值需要X的全部信息才能确定，去掉X中任何一个属性就不行了。

**示例**：在选课表(学号, 课程号, 成绩)中，(学号, 课程号)→成绩是完全函数依赖，因为只靠学号或只靠课程号都不能确定成绩。

**（2）部分函数依赖**

**定义**：如果X→Y，但Y不完全函数依赖于X，则称Y对X部分函数依赖，记作 X ——P→ Y。

**通俗解释**：Y的值只需要X的一部分信息就能确定，不需要X的全部。

**示例**：在(学号, 课程号, 姓名)中，(学号, 课程号)→姓名是部分函数依赖，因为只靠学号就能确定姓名，不需要课程号。

**（3）传递函数依赖**

**定义**：在R(U)中，如果X→Y，Y→Z，且Y→X不成立（Y不是X的子集，即Y不包含在X中），Z∉Y，则称Z对X传递函数依赖。

**通俗解释**：X决定Y，Y决定Z，但Y不能反过来决定X（Y不是候选码），所以Z通过Y传递依赖于X。

**示例**：在(学号, 系名, 系主任)中，学号→系名，系名→系主任，且系名→学号不成立，所以系主任传递依赖于学号。

### 5.1.3 三种函数依赖对比

| 类型 | 符号 | 含义 | 关键特征 |
|------|------|------|---------|
| 完全函数依赖 | X ——F→ Y | Y完全依赖于X | 需要X的全部属性 |
| 部分函数依赖 | X ——P→ Y | Y部分依赖于X | 只需要X的部分属性 |
| 传递函数依赖 | X → Y → Z | Z传递依赖于X | 通过中间属性Y传递 |

### 【例题1】（2018年上半年）

**题目**：设有关系模式R(A, B, C, D)，函数依赖集F={A→B, B→C, A→D}。则C对A的函数依赖是（）。
A. 完全函数依赖
B. 部分函数依赖
C. 传递函数依赖
D. 不是函数依赖

**答案**：C

**解析**：A→B，B→C，所以C传递依赖于A。

### 【例题2】（2020年下半年）

**题目**：设有关系模式R(学号, 课程号, 成绩, 姓名)，其中(学号, 课程号)是主码。则以下说法正确的是（）。
A. 成绩对(学号, 课程号)是完全函数依赖
B. 姓名对(学号, 课程号)是完全函数依赖
C. 成绩对学号是完全函数依赖
D. 姓名对课程号是完全函数依赖

**答案**：A

**解析**：成绩需要同时知道学号和课程号才能确定，所以成绩对(学号, 课程号)是完全函数依赖。姓名只需要学号就能确定，所以姓名对(学号, 课程号)是部分函数依赖。

## 5.2 码的求解方法（★ 高频计算）

### 5.2.1 候选码的定义

**教材定义**：设K为R(U,F)中的属性或属性组合，若K →F U（K完全函数决定U），则K为R的候选码。

**通俗解释**：候选码就是能唯一决定所有属性的最小属性集。候选码之外的属性不能函数推导出候选码中的任何属性。

### 5.2.2 候选码求解算法（★★★）

**步骤**：
1. **找只出现在函数依赖左边的属性**：这些属性一定在候选码中
2. **找只出现在函数依赖右边的属性**：这些属性一定不在候选码中
3. **找既出现在左边又出现在右边的属性**：这些属性可能在候选码中
4. **合并**：用"只在左边"的属性尝试推导所有属性：
   - 如果能推导出所有属性，则这就是一个候选码
   - 如果不能，再依次加入"既在左边又在右边"的属性尝试
5. **检验最小性**：确保候选码的任何真子集都不能推导出所有属性

**示例**：
设有关系模式R(A, B, C, D, E)，F={A→B, C→D, A→E}。

步骤1：只在左边的属性：A, C（因为A和C只出现在→的左边）
步骤2：只在右边的属性：B, D, E
步骤3：没有既在左边又在右边的属性
步骤4：先用AC推导：AC→ABCDE（A→B, A→E, C→D），能推导出所有属性！
步骤5：检验最小性：A不能推导所有属性（缺C, D），C不能推导所有属性（缺A, B, E）。所以AC是最小的。
**候选码 = {A, C}**
**主属性 = {A, C}**，**非主属性 = {B, D, E}**

### 【例题1】（2019年下半年）

**题目**：设有关系模式R(A, B, C, D)，F={A→B, B→C, D→B}。求R的所有候选码。

**解答**：
- 只在左边：A, D
- 只在右边：C
- 既左又右：B
- 先用AD推导：AD→ABD，不能推导出C
- 再试：AD+B？但B不在候选码中（因为候选码要最小）
- 实际上AD推导：A→B→C，D→B。所以AD→ABCD。
- 检验：A不能推导D，D不能推导A
- **候选码 = {A, D}**

### 【例题2】（2021年上半年）

**题目**：设有关系模式R(U,F)，U={A,B,C,D,E}，F={A→B, BC→D, D→E}。求R的候选码。

**解答**：
- 只在左边：A, C（A只在左边，C只在BC的左边）
- 只在右边：E（E只在D→E的右边）
- 既左又右：B, D
- 先用AC推导：A→B，ABC→D（BC→D），ABCDE→E（D→E）。
- 所以AC→ABCDE。
- 检验最小性：A不能推导C，C不能推导A。
- **候选码 = {A, C}**

### 【例题3】（2022年下半年）

**题目**：设有关系R(A, B, C, D, E)，F={A→B, C→D, AB→E}。求R的候选码和主属性。

**解答**：
- 只在左边：A, C（A和C只出现在左边）
- 只在右边：B, D, E
- 没有既在左边又在右边的
- AC推导：A→B，C→D，AB→E。所以AC→ABCDE
- 检验最小性：A不推导C、D，C不推导A、B、E
- **候选码 = {A, C}**
- **主属性 = {A, C}**，**非主属性 = {B, D, E}**

## 5.3 范式

### 5.3.1 概述

**教材定义**：关系数据库中的关系是要满足一定要求的，满足不同程度要求的为不同范式。满足最低要求的叫第一范式（1NF），在1NF基础上满足进一步要求的为第二范式（2NF），以此类推。

**通俗解释**：范式就像"数据库设计的质量标准"——从1NF到BCNF，标准越来越高，数据冗余越来越少。

### 5.3.2 第一范式（1NF）

**定义**：关系模式R的每一个属性都是不可再分的数据项，则R∈1NF。

**通俗解释**：每个格子只能放一个值，不能放"一堆值"。表中不能再有表。

**示例**：
```
不满足1NF：
| 学号 | 姓名 | 电话号码        |
|------|------|----------------|
| 001  | 张三 | 138xxxx, 139xxxx  |  ← 一个格子放了两个值

满足1NF：
| 学号 | 姓名 | 电话号码        |
|------|------|----------------|
| 001  | 张三 | 138xxxx         |
| 001  | 张三 | 139xxxx         |
```

### 5.3.3 第二范式（2NF）

**定义**：R∈1NF，且R中的每一个非主属性都完全函数依赖于任何一个候选码，则R∈2NF。

**通俗解释**：在1NF的基础上，消除部分依赖。非主属性必须完全依赖于主码，不能只依赖于主码的一部分。

**示例**：
```
关系模式SC(学号, 课程号, 成绩, 姓名)，主码=(学号, 课程号)
- 成绩完全依赖于(学号, 课程号) ✓
- 姓名部分依赖于(学号, 课程号)（只依赖学号）✗
- 不满足2NF

分解：
- SC(学号, 课程号, 成绩)  ← 满足2NF
- Student(学号, 姓名)      ← 满足2NF
```

### 5.3.4 第三范式（3NF）

**定义**：R∈1NF，且R中不存在这样的候选码X、属性组Y及非主属性Z（Z∉Y），使得X→Y，Y→Z成立，Y→X不成立。简言之：不存在非主属性对候选码的传递依赖。

**通俗解释**：在2NF的基础上，消除传递依赖。非主属性不能通过中间属性间接依赖于主码。

**示例**：
```
关系模式(学号, 系名, 系主任)，主码=学号
- 学号→系名，系名→系主任
- 系主任传递依赖于学号
- 不满足3NF

分解：
- (学号, 系名)  ← 满足3NF
- (系名, 系主任) ← 满足3NF
```

### 5.3.5 BCNF（Boyce-Codd范式）

**定义**：R∈1NF，若X→Y且Y∉X时X必含有候选码，则R∈BCNF。

**通俗解释**：在3NF的基础上，消除主属性的部分依赖和传递依赖。所有决定因素（箭头左边）都必须包含候选码。

**另一种说法**：每个函数依赖的决定因素都必须是超键。

**示例**：
```
关系模式(S, T, J)，S=学生，T=教师，J=课程
假设：每个教师只教一门课，每门课有多个教师
函数依赖：(S, J)→T, (S, T)→J, T→J
候选码：(S, J) 和 (S, T)
- 3NF检查：没有非主属性（所有属性都是主属性）→ 满足3NF
- BCNF检查：T→J，T不是候选码 → 不满足BCNF

分解：
- (T, J)  ← 满足BCNF
- (S, T)  ← 满足BCNF
```

### 5.3.6 各范式关系总结

```
1NF ⊃ 2NF ⊃ 3NF ⊃ BCNF
```

| 范式 | 消除的问题 | 条件 |
|------|-----------|------|
| 1NF | 消除重复组/复合属性 | 属性不可再分 |
| 2NF | 消除非主属性的部分函数依赖 | 非主属性完全依赖于候选码 |
| 3NF | 消除非主属性的传递函数依赖 | 非主属性不传递依赖于候选码 |
| BCNF | 消除主属性的部分和传递依赖 | 每个决定因素都含候选码 |

**判断顺序**：
1. 是否1NF？（属性是否不可分）
2. 是否2NF？（有无非主属性的部分依赖）
3. 是否3NF？（有无非主属性的传递依赖）
4. 是否BCNF？（有无主属性的部分/传递依赖）

### 【例题1】（2018年下半年）

**题目**：设有关系模式R(学号, 课程号, 成绩, 姓名, 系名)，主码为(学号, 课程号)。函数依赖有：学号→姓名，学号→系名，(学号, 课程号)→成绩。R最高满足（）。
A. 1NF
B. 2NF
C. 3NF
D. BCNF

**答案**：A

**解析**：R中非主属性姓名、系名对主码(学号, 课程号)是部分函数依赖（只需要学号），所以R不满足2NF，只满足1NF。

### 【例题2】（2020年上半年）

**题目**：关系模式R中，如果所有非主属性都完全函数依赖于候选码，且不存在传递依赖，则R满足（）。
A. 1NF
B. 2NF
C. 3NF
D. BCNF

**答案**：C

**解析**：所有非主属性完全函数依赖于候选码 = 满足2NF。不存在传递依赖 = 满足3NF。

### 【例题3】（2021年下半年）

**题目**：BCNF要求（）。
A. 消除非主属性对候选码的部分依赖
B. 消除非主属性对候选码的传递依赖
C. 消除主属性对候选码的部分和传递依赖
D. 每个决定因素都包含候选码

**答案**：D

**解析**：BCNF要求每个函数依赖的决定因素（箭头左边）都必须包含候选码（即必须是超键）。

### 【例题4】（2023年上半年）

**题目**：设有关系模式R(A, B, C, D)，F={A→B, B→C, A→D}，候选码为A。则R满足（）范式。
A. 1NF
B. 2NF
C. 3NF
D. BCNF

**答案**：B

**解析**：候选码为A，非主属性为B、C、D。A→B是完全依赖（满足2NF）。但B→C，所以C传递依赖于A（A→B→C），不满足3NF。所以最高满足2NF。

## 5.4 模式分解

### 5.4.1 无损连接分解

**定义**：设关系模式R分解为若干个子模式R1, R2, ..., Rk，如果R的任意一个关系r都满足 r = π_{R1}(r) ⋈ π_{R2}(r) ⋈ ... ⋈ π_{Rk}(r)，则称该分解具有无损连接性。

**通俗解释**：分解后能通过自然连接恢复原关系，不丢失任何信息。

**判定定理（★★★）**：
R分解为R1和R2，该分解具有无损连接性的充要条件是：
R1 ∩ R2 → R1 − R2 或 R1 ∩ R2 → R2 − R1
即：两个子模式的公共属性必须能函数决定其中至少一个子模式的全部属性。

**示例**：
R(A, B, C, D, E)，F={A→B, C→D}，分解为R1(A, B, C)和R2(C, D, E)。
- R1 ∩ R2 = {C}
- R1 − R2 = {A, B}，C→A,B 不成立
- R2 − R1 = {D, E}，C→D 成立（C→D在F中）
- 该分解是无损连接分解 ✓

### 5.4.2 保持函数依赖分解

**定义**：设F是关系模式R的函数依赖集，分解后的子模式的函数依赖集分别为F1, F2, ..., Fk。如果F1∪F2∪...∪Fk与F等价，则称该分解保持函数依赖。

**通俗解释**：分解后各个子模式的函数依赖集的并集，能推导出原始的全部函数依赖，不丢失约束信息。

### 5.4.3 无损连接 vs 保持函数依赖

| 对比项 | 无损连接 | 保持函数依赖 |
|--------|---------|-------------|
| 含义 | 分解后能恢复原关系 | 分解后不丢失函数依赖 |
| 重要性 | 必须满足 | 尽量满足 |
| 判定方法 | 矩阵法/定理法 | 检查F1∪F2∪...∪Fk是否等价于F |

### 【例题1】（2019年上半年）

**题目**：关系模式R(A, B, C)，F={A→B, B→C}，分解为R1(A, B)和R2(B, C)。判断该分解是否是无损连接分解。

**解答**：
- R1 ∩ R2 = {B}
- R1 − R2 = {A}，B→A 不成立
- R2 − R1 = {C}，B→C 成立
- 该分解是无损连接分解 ✓

### 【例题2】（2022年上半年）

**题目**：关系模式R(A, B, C, D)，F={A→B, C→D}，分解为R1(A, B)和R2(C, D)。判断该分解是否是无损连接分解。

**解答**：
- R1 ∩ R2 = ∅（空集）
- R1 − R2 = {A, B}，∅→AB 不成立
- R2 − R1 = {C, D}，∅→CD 不成立
- 该分解不是无损连接分解 ✗

**解析**：公共属性为空集，无法满足无损连接条件。

### 【学习提示】

- **考试技巧**：范式判断和码的求解是上午选择题必考内容（2-4分）
- **判断范式的步骤**：先求候选码→分主属性/非主属性→检查部分依赖→检查传递依赖
- **下午题**：通常会结合E-R图考规范化理论，要求分解到3NF
- **无损连接判定**：记住公式——公共属性至少能决定其中一个差集

# 第六章 数据库设计（★★★ 下午题必考 15 分）

## 6.1 数据库设计的六个阶段

### 6.1.1 总体概述

**教材定义**：数据库设计是指对于一个给定的应用环境，构造（设计）优化的数据库逻辑模式和物理结构，并据此建立数据库及其应用系统，使之能够有效地存储和管理数据，满足各种用户的应用需求。

**通俗解释**：数据库设计就像盖房子——先了解需求（要什么样的房子），再画图纸（概念设计），再定结构（逻辑设计），再选材料和施工方式（物理设计），然后施工（实施），最后维护。

### 6.1.2 六个阶段详解

| 阶段 | 主要任务 | 输出成果 |
|------|---------|---------|
| 需求分析 | 了解用户需求，收集数据流图、数据字典 | 数据流图、数据字典 |
| 概念结构设计 | 设计E-R图 | E-R图（全局E-R图） |
| 逻辑结构设计 | E-R图→关系模式，规范化处理 | 关系模式（表结构） |
| 物理结构设计 | 选择存储结构和存取方法 | 物理设计说明书 |
| 数据库实施 | 建库、加载数据、编写程序、测试 | 可运行的数据库系统 |
| 数据库运行和维护 | 运行监控、性能调优、备份恢复 | 维护记录 |

**详细说明**：

**(1) 需求分析阶段**

这是数据库设计的第一步，也是最重要的一步。设计人员需要充分调查用户的需求，收集用户所需要处理的数据，了解用户对数据的安全性和完整性要求。主要工作包括：
- 调查组织机构情况
- 调查各部门的业务活动情况
- 明确新系统的各种要求
- 确定系统的边界

输出成果：数据流图（DFD）和数据字典（DD）。

**(2) 概念结构设计阶段**

对用户需求进行综合、归纳与抽象，形成一个独立于具体DBMS的概念模型。最常用的设计方法是E-R模型。主要工作包括：
- 抽象数据并设计局部E-R图
- 合并局部E-R图，消除冲突（属性冲突、命名冲突、结构冲突）
- 消除冗余，生成基本E-R图

输出成果：E-R图。

**(3) 逻辑结构设计阶段**

将概念结构转换为某个DBMS所支持的数据模型（主要是关系模型），并对转换后的模型进行优化。主要工作包括：
- E-R图→关系模式
- 数据模型的优化
- 设计用户子模式（视图）

输出成果：关系模式。

**(4) 物理结构设计阶段**

为逻辑数据模型选取一个最适合应用环境的物理结构（存储结构和存取方法）。主要工作包括：
- 确定数据库的存储结构
- 设计合适的存取路径（如索引）
- 评价物理结构的时间和空间效率

输出成果：物理设计说明书。

**(5) 数据库实施阶段**

根据逻辑设计和物理设计的结果，在计算机上建立实际的数据库结构，装入数据，测试和试运行。主要工作包括：
- 定义数据库结构
- 组织数据入库
- 编制与调试应用程序
- 数据库试运行

**(6) 数据库运行和维护阶段**

数据库系统经过试运行后投入正式运行，在运行过程中需要不断评价、调整与修改。主要工作包括：
- 数据库的转储和恢复
- 数据库的安全性、完整性控制
- 数据库性能的监督、分析和改造
- 数据库的重组织和重构造

### 【例题1】（2019年上半年）

**题目**：数据库设计中，概念结构设计阶段的主要成果是（）。
A. 数据流图
B. E-R图
C. 关系模式
D. 物理设计说明书

**答案**：B

**解析**：概念结构设计阶段的成果是E-R图。数据流图是需求分析阶段的成果，关系模式是逻辑结构设计阶段的成果，物理设计说明书是物理设计阶段的成果。

**题目**：数据库设计中，将E-R图转换为关系模式属于（）阶段。
A. 需求分析
B. 概念结构设计
C. 逻辑结构设计
D. 物理结构设计

**答案**：C

**解析**：E-R图转换为关系模式是逻辑结构设计阶段的工作。

### 【例题3】（2022年上半年 下午题）

**题目**：某学校有多个学院，每个学院有多名教师。教师有工号、姓名、职称等属性。学院有学院号、学院名、院长等属性。一名教师只能属于一个学院，一个学院有一名院长（从教师中选）。
（1）画出E-R图。
（2）将E-R图转换为关系模式，并指出主码和外码。

**解答**：
(1) E-R图：
- 实体：学院(学院号, 学院名)，教师(工号, 姓名, 职称)
- 联系：隶属(1:n)，领导(1:1)

(2) 关系模式：
- 学院(学院号, 学院名, 院长工号)，主码：学院号，外码：院长工号参照教师(工号)

**解析**：教师与学院是1:n联系（一名教师属于一个学院），合并到n端（教师表加学院号）。学院与院长是1:1联系（一个学院一个院长），合并到学院表（加院长工号）。

## 6.2 需求分析

### 6.2.1 主要内容

需求分析的重点是调查、收集和分析用户在数据管理中的信息要求、处理要求、安全性与完整性要求。

| 需求类型 | 说明 |
|---------|------|
| 信息要求 | 用户需要从数据库中获得什么信息 |
| 处理要求 | 用户要完成什么处理功能 |
| 安全性要求 | 数据的保密程度和存取控制要求 |
| 完整性要求 | 数据的有效性、正确性和相容性要求 |

### 6.2.2 数据流图（DFD）

数据流图是描述系统数据流程的工具，它用抽象的符号描述数据在系统中的流动、处理和存储情况。

### 6.2.3 数据字典（DD）

数据字典是关于数据库中数据的描述，是各类数据描述的集合。数据字典通常包括：
- 数据项（最小数据单位）
- 数据结构（数据项的组合）
- 数据流（数据结构在系统内的传输路径）
- 数据存储（数据结构停留或保存的地方）
- 处理过程（处理逻辑）

## 6.3 概念结构设计

### 6.3.1 E-R 图的设计方法

**方法一：自顶向下**：首先定义全局概念结构的框架，然后逐步细化。

**方法二：自底向上（最常用）**：首先定义各局部应用的概念结构（局部E-R图），然后将它们集成起来，得到全局概念结构（全局E-R图）。

**方法三：逐步扩张**：首先定义最重要的核心概念结构，然后向外扩充。

**方法四：混合策略**：将自顶向下和自底向上相结合。

### 6.3.2 局部E-R图的设计

1. 选择局部应用
2. 逐一设计局部E-R图
3. 确定实体和属性
4. 确定实体之间的联系

### 6.3.3 全局E-R图的集成

将多个局部E-R图合并为全局E-R图时，需要解决以下**冲突**：

| 冲突类型 | 说明 | 解决方法 |
|---------|------|---------|
| 属性冲突 | 同一属性在不同局部E-R图中的类型、取值范围不同 | 统一属性的类型和取值范围 |
| 命名冲突 | 同名异义或异名同义 | 统一命名 |
| 结构冲突 | 同一对象在不同局部E-R图中的抽象级别不同 | 统一抽象级别 |

### 6.3.4 消除冗余

冗余的数据和冗余的联系应该消除。冗余数据是指能由基本数据导出的数据，冗余联系是指能由其他联系导出的联系。

## 7.1 事务的概念与 ACID 特性

### 7.1.1 事务的定义

**教材定义**：事务是用户定义的一个数据库操作序列，这些操作要么全做，要么全不做，是一个不可分割的工作单位。

**通俗解释**：事务就像"打包操作"——一组操作要么全成功（提交），要么全失败（回滚）。比如转账：A扣100元、B加100元，要么都成功，要么都失败，不能只扣不加。

**详细说明**：在关系数据库中，一条SQL语句就是一个事务（自动提交模式下）。但实际应用中，往往需要将多条SQL语句组合成一个事务。事务以BEGIN TRANSACTION开始，以COMMIT（提交）或ROLLBACK（回滚）结束。

### 7.1.2 ACID 特性

| 特性 | 英文 | 含义 | 通俗解释 |
|------|------|------|---------|
| 原子性 | Atomicity | 事务中包含的各项操作，要么全部执行，要么全部不执行 | "全做或全不做" |
| 一致性 | Consistency | 事务执行的结果必须是使数据库从一个一致性状态变到另一个一致性状态 | "状态守恒"——转账前后总金额不变 |
| 隔离性 | Isolation | 一个事务的执行不能被其他事务干扰 | "互不干扰"——并行事务就像串行执行一样 |
| 持久性 | Durability | 事务一旦提交，它对数据库中数据的改变就应该是永久性的 | "提交了就不回头" |

**详细说明**：
- **原子性**由DBMS的**恢复管理子系统**实现。如果事务执行过程中发生故障，恢复管理子系统会撤销（UNDO）事务已做的操作，使数据库恢复到事务开始时的状态。
- **一致性**是事务的根本目的。其他三个特性都是为了保证一致性而存在的。
- **隔离性**由DBMS的**并发控制子系统**实现。通过封锁等机制保证并发执行的事务互不干扰。
- **持久性**由DBMS的**恢复管理子系统**实现。一旦事务提交，即使系统发生故障，也能保证提交的结果不会丢失。

### 【例题1】（2018年上半年）

**题目**：事务的ACID特性中，C表示（）。
A. 原子性
B. 一致性
C. 隔离性
D. 持久性

**答案**：B

**解析**：ACID分别代表Atomicity（原子性）、Consistency（一致性）、Isolation（隔离性）、Durability（持久性）。C对应Consistency（一致性）。

### 【例题2】（2020年下半年）

C. 一个事务的执行不能被其他事务干扰
D. 事务一旦提交，对数据库的改变就是永久性的

**答案**：B

**解析**：A是原子性，B是一致性，C是隔离性，D是持久性。

### 【例题3】（2022年上半年）

**题目**：银行转账业务中，A向B转账100元。从数据库事务的角度来看，以下说法正确的是（）。

**解析**：转账是一个事务，包含A减100和B加100两个操作，要么都执行，要么都不执行（原子性）。C错误，如果分开成两个事务，就可能出现A减了但B没加的情况。D错误，持久性保证的是提交后不丢失，不保证事务一定能成功。

## 7.2 并发控制

### 7.2.1 并发操作带来的问题

**教材定义**：当多个用户并发地存取数据库时，就会产生多个事务同时存取同一数据的情况，若对并发操作不加控制，就可能会存取和存储不正确的数据，破坏事务的一致性和数据库的一致性。

**通俗解释**：多个事务同时操作同一数据，如果不加控制，就会出乱子。

### 7.2.2 三类数据不一致性

| 类型 | 英文 | 说明 | 示例 |
|------|------|------|------|
| 丢失更新 | Lost Update | 两个事务同时读取并修改同一数据，后写的覆盖了先写的 | T1写A=10，T2写A=20，T1的更新丢失 |
| 不可重复读 | Non-repeatable Read | 事务两次读取同一数据，结果不一致（因为其他事务修改了数据） | T1读A=10，T2修改A=20并提交，T1再读A=20 |
| 读脏数据 | Dirty Read | 事务读取了其他事务未提交的数据，后来那个事务回滚了 | T1修改A=20（未提交），T2读A=20，T1回滚（A恢复10） |

**详细说明**：
- **丢失更新**：T1和T2同时读A=10，T1修改A=20写回，T2修改A=30写回。T1的更新被T2覆盖了，T1的更新丢失。
- **不可重复读**：T1读A=10后，T2修改A=20并提交，T1再读A=20，两次读的结果不一致。
- **读脏数据**：T1修改A=10→20但未提交，T2读到A=20，之后T1回滚（A恢复为10），T2读到的数据是无效的"脏"数据。

### 7.2.3 三类不一致对比

| 对比项 | 丢失更新 | 不可重复读 | 读脏数据 |
|--------|---------|-----------|---------|
| 原因 | 两个事务同时写 | 一个读一个写 | 读了未提交的数据 |
| 后果 | 后写的覆盖先写的 | 两次读结果不同 | 读到可能被撤销的数据 |
| 读写关系 | 写-写冲突 | 读-写冲突 | 读-写冲突 |

### 【例题1】（2019年上半年）

**题目**：事务T1读取了事务T2修改但未提交的数据，如果T2回滚，则T1读到的是（）。
A. 丢失更新
B. 不可重复读
C. 读脏数据
D. 幻影读

**答案**：C

**解析**：读了其他事务未提交的数据，后来那个事务回滚了，这叫"读脏数据"。

### 【例题2】（2021年下半年）

**题目**：以下哪种情况属于"丢失更新"？（）
A. T1读A，T2修改A，T1再读A结果变了
B. T1修改A未提交，T2读A
C. T1和T2同时读A=10，各自修改后写回，先写的被覆盖
D. T1删除了一条记录，T2查询时发现记录减少了

**答案**：C

**解析**：丢失更新是指两个事务同时读取并修改同一数据，后写的覆盖了先写的。C描述的就是这种情况。

## 7.3 封锁协议

### 7.3.1 封锁的基本概念

**教材定义**：封锁就是事务T在对某个数据对象（表、记录等）操作之前，先向系统发出请求，对其加锁。加锁后事务T就对该数据对象有了一定的控制，在事务T释放它的锁之前，其他事务不能更新此数据对象。

**通俗解释**：封锁就像"上锁"——我要操作这个数据，先锁起来，不让别人动。

**两种基本锁**：

| 锁类型 | 英文 | 相容性 | 说明 |
|--------|------|--------|------|
| 排他锁（X锁） | eXclusive Lock | 不与其他锁相容 | 又称写锁。事务T对数据加了X锁，其他事务不能再加任何锁 |
| 共享锁（S锁） | Shared Lock | 只与S锁相容 | 又称读锁。事务T对数据加了S锁，其他事务可以加S锁但不能加X锁 |

### 7.3.2 三级封锁协议

| 等级 | 规则 | 解决的问题 |
|------|------|-----------|
| 一级封锁协议 | 修改前必须加X锁，事务结束才释放 | 丢失更新 |
| 二级封锁协议 | 在一级基础上，读取前加S锁，读完即释放 | 丢失更新 + 读脏数据 |
| 三级封锁协议 | 在一级基础上，读取前加S锁，事务结束才释放 | 丢失更新 + 读脏数据 + 不可重复读 |

**详细说明**：
- **一级封锁协议**：事务在修改数据之前必须先加X锁，直到事务结束（COMMIT或ROLLBACK）才释放。可以防止丢失更新，但不能防止读脏数据和不可重复读。
- **二级封锁协议**：在一级的基础上，事务在读取数据之前必须先加S锁，读完即可释放S锁。可以防止丢失更新和读脏数据，但不能防止不可重复读。
- **三级封锁协议**：在一级的基础上，事务在读取数据之前必须先加S锁，直到事务结束才释放S锁。可以防止丢失更新、读脏数据和不可重复读。

### 7.3.3 两段锁协议（2PL）

**教材定义**：所有事务必须分两个阶段对数据项加锁和解锁：
- **扩展阶段（Growing Phase）**：事务可以获得锁，但不能释放锁
- **收缩阶段（Shrinking Phase）**：事务可以释放锁，但不能获得锁

**通俗解释**：先拼命"拿锁"（扩展阶段），然后才开始"还锁"（收缩阶段），不能边拿边还。

**重要结论**：
- 遵守两段锁协议的事务是可串行化的（可以并发执行且结果与某种串行执行结果等价）
- 可串行化是并发控制的正确性准则
- 两段锁协议是保证并发调度可串行化的充分条件，不是必要条件

### 【例题1】（2018年下半年）

**题目**：一级封锁协议可以解决的问题是（）。
A. 丢失更新
B. 读脏数据
C. 不可重复读
D. 幻影读

**答案**：A

**解析**：一级封锁协议规定修改前加X锁，事务结束才释放，可以防止丢失更新。二级协议可以防止读脏数据，三级协议可以防止不可重复读。

### 【例题2】（2020年上半年）

**题目**：两段锁协议是指（）。
A. 事务分为两个阶段，第一阶段加锁，第二阶段解锁

**答案**：C

**解析**：两段锁协议规定：扩展阶段（第一阶段）只加锁不解锁，收缩阶段（第二阶段）只解锁不加锁。

### 【例题3】（2023年上半年）

## 7.4 活锁与死锁

### 7.4.1 活锁（Livelock）

**定义**：如果事务T1封锁了数据R，事务T2请求封锁R于是等待，T3也请求封锁R。当T1释放R上的锁时，系统批准了T3的请求，T2继续等待。然后T4又请求封锁R，当T3释放锁时，系统又批准了T4的请求……T2可能永远等待。

**通俗解释**：排队时一直有人插队到你前面，你永远排不上。

**解决方法**：采用**先来先服务（FCFS）**策略。

### 7.4.2 死锁（Deadlock）

**定义**：如果事务T1封锁了数据R1，T2封锁了数据R2，然后T1又请求封锁R2（需要等T2释放），T2又请求封锁R1（需要等T1释放），于是两个事务永远互相等待。

**通俗解释**：两个人互相等对方让路，谁也不让，就僵住了。

**死锁的预防**：

| 方法 | 说明 |
|------|------|
| 一次封锁法 | 事务开始时一次性封锁所有需要的数据 |
| 顺序封锁法 | 规定封锁数据对象的顺序，所有事务按这个顺序封锁 |

**死锁的诊断与解除**：

| 方法 | 说明 |
|------|------|
| 超时法 | 事务等待超过规定时间就认为死锁，回滚该事务 |
| 等待图法 | 构造事务等待图，检测是否有环。有环则死锁 |

### 7.4.3 活锁与死锁对比

| 对比项 | 活锁 | 死锁 |
|--------|------|------|
| 表现 | 事务永远等待但系统仍在运行 | 事务互相等待，永远无法继续 |
| 原因 | 锁分配不公平 | 事务间形成循环等待 |
| 解决方法 | FCFS | 预防或检测解除 |

### 【例题1】（2019年下半年）

**题目**：在数据库并发控制中，死锁是指（）。
A. 一个事务永远等待锁
B. 两个或多个事务互相等待对方释放锁
C. 事务执行超时
D. 事务被回滚

**答案**：B

**解析**：死锁是两个或多个事务互相等待对方释放锁，形成循环等待。A描述的是活锁。

### 【例题2】（2021年上半年）

**题目**：预防死锁的方法中，一次封锁法是指（）。
A. 一次只封锁一个数据对象

**答案**：B

**解析**：一次封锁法要求事务在开始执行前，一次性封锁所有需要的数据对象。这样可以避免死锁，但会降低并发度。

# 第八章 数据库备份与恢复

## 8.1 故障类型

### 8.1.1 三类故障

| 故障类型 | 说明 | 恢复方法 |
|---------|------|---------|
| 事务故障 | 某个事务在运行过程中由于各种原因未运行至正常终止点就夭折了 | UNDO（撤销）该事务已做的所有修改 |
| 系统故障 | 造成系统停止运转的任何事件，使得系统需要重新启动（如停电、操作系统崩溃） | REDO（重做）已提交的事务，UNDO未完成的事务 |
| 介质故障 | 外存故障，如磁盘损坏、磁头碰撞、病毒破坏等 | 装入最近的数据库后备副本，然后重做（REDO）已完成的事务 |

**通俗解释**：
- 事务故障：一个操作没做完就出错了 → 撤销回去
- 系统故障：整个系统挂了 → 恢复到最近的正确状态
- 介质故障：硬盘坏了 → 从备份恢复

**详细说明**：
- **事务故障**是非预期的、不可预料的，包括：运算溢出、违反完整性约束、并发事务发生死锁而被选中撤销、应用程序错误等。恢复由系统自动完成，对用户透明。
- **系统故障**也称为"软故障"。发生系统故障时，内存中数据库缓冲区的数据全部丢失，但存储在外存上的数据库未被破坏。恢复时需要：(1) 正向扫描日志文件，建立重做队列和撤销队列；(2) 对撤销队列中的事务进行UNDO处理；(3) 对重做队列中的事务进行REDO处理。
- **介质故障**也称为"硬故障"。介质故障会破坏数据库，需要从备份恢复。

### 【例题1】（2019年下半年）

**题目**：系统故障的恢复需要（）。
A. 只UNDO
B. 只REDO
C. UNDO未完成事务 + REDO已提交事务
D. 从备份恢复

**答案**：C

**解析**：系统故障时，内存数据丢失但外存数据完好。恢复时需要：UNDO未完成的事务（撤回未提交的修改），REDO已提交的事务（重做已提交的修改，因为修改可能还在缓冲区没写入磁盘）。

### 【例题2】（2022年下半年）

**题目**：介质故障是指（）。
A. 事务执行中的逻辑错误
B. 系统崩溃需要重启
C. 外存设备损坏导致数据丢失
D. 并发事务死锁

**答案**：C

**解析**：介质故障是指外存设备（如硬盘）损坏导致数据丢失，需要从备份恢复。A是事务故障，B是系统故障，D是并发控制问题。

## 8.2 备份与恢复技术

### 8.2.1 数据转储（备份）

| 转储方式 | 说明 | 优缺点 |
|---------|------|--------|
| 静态转储 | 在系统中无运行事务时进行转储 | 简单，但会降低系统可用性 |
| 动态转储 | 转储期间允许事务运行 | 不影响系统运行，但需要配合日志 |
| 海量转储 | 转储全部数据库 | 恢复简单，但耗时长 |
| 增量转储 | 只转储上次转储后更新过的数据 | 节省时间和空间，但恢复较复杂 |

**组合方式**：
- 静态海量转储
- 静态增量转储
- 动态海量转储
- 动态增量转储

### 8.2.2 数据库恢复

**恢复策略**：
1. 装入最新的数据库后备副本
2. 装入转储后的日志文件
3. 对日志文件中的已提交事务做REDO
4. 对未完成的事务做UNDO

### 【例题1】（2020年上半年）

**题目**：在数据库恢复技术中，增量转储是指（）。
A. 转储全部数据库
B. 只转储上次转储后更新过的数据
C. 在系统无运行事务时转储
D. 转储日志文件

**答案**：B

**解析**：增量转储是指只转储上次转储后更新过的数据。海量转储是指转储全部数据库。静态转储是在无运行事务时转储。

## 8.3 日志文件的作用

### 8.3.1 日志文件的定义

**教材定义**：日志文件是用来记录事务对数据库的更新操作的文件。

**通俗解释**：日志就像"操作记录本"——记录谁在什么时候对什么数据做了什么修改。

### 8.3.2 日志文件的内容

每个日志记录通常包含：
- 事务标识（标明是哪个事务）
- 操作类型（插入、删除、修改）
- 操作对象
- 更新前的数据值（旧值/BFIM）
- 更新后的数据值（新值/AFIM）

### 8.3.3 日志文件的作用

1. **事务恢复**：根据日志进行UNDO和REDO
2. **数据库恢复**：介质故障时配合备份进行恢复
3. **审计追踪**：记录数据库的操作历史

### 8.3.4 恢复的原则

1. **先写日志，后修改数据库**：确保日志记录在数据库修改之前写入稳定存储
2. 已提交事务的更新不应丢失
3. 未提交事务的更新应被撤销

### 【例题1】（2018年下半年）

**题目**：数据库日志文件的主要作用是（）。
A. 存储用户数据
B. 记录事务对数据库的更新操作，用于恢复
C. 定义数据库结构
D. 实现并发控制

**答案**：B

**解析**：日志文件记录了事务对数据库的所有更新操作，是数据库恢复的重要依据。

### 【例题2】（2021年上半年）

**题目**：数据库恢复的基本原则是（）。
A. 先修改数据库，后写日志
B. 先写日志，后修改数据库
C. 同时写日志和修改数据库
D. 不需要写日志

**答案**：B

**解析**：数据库恢复的基本原则是"先写日志文件"（Write-Ahead Logging），即在修改数据库之前，先将修改记录写入日志文件，确保即使在修改过程中发生故障，也能通过日志进行恢复。

# 第九章 数据库安全性

---

## 9.1 用户标识与鉴别

### 9.1.1 概述

**教材定义**：用户标识与鉴别是系统提供的最外层安全保护措施。每次用户要求进入系统时，由系统进行核对，通过鉴定后才提供使用权。

**通俗解释**：就是"登录验证"——用户名+密码，确认你是谁。

**常见的用户标识与鉴别方法**：
- 用户名/口令（最常用）
- 磁卡/IC卡
- 生物特征（指纹、虹膜等）

### 【例题1】（2019年上半年）

**题目**：数据库安全性控制的最外层措施是（）。
A. 用户标识与鉴别
B. 存取控制
C. 视图机制
D. 数据加密

**答案**：A

**解析**：用户标识与鉴别是系统提供的最外层安全保护措施，即"登录验证"。

---

## 9.2 存取控制

### 9.2.1 概述

**教材定义**：存取控制机制的功能是确保只授权给有资格的用户访问数据库的权限，同时令所有未被授权的人员无法接近数据。

**通俗解释**：存取控制就是"权限管理"——谁能看什么、谁能改什么。

### 9.2.2 DAC 与 MAC

| 对比项 | DAC（自主存取控制） | MAC（强制存取控制） |
|--------|-------------------|-------------------|
| 定义 | 用户可以自主地决定将自己拥有的存取权限授予其他用户 | 由系统强制实施的存取控制，用户不能改变 |
| 粒度 | 较粗（表级、列级） | 较细（数据对象有密级，用户有许可证级别） |
| 灵活性 | 高 | 低 |
| 安全性 | 较低 | 较高 |
| 适用场景 | 一般应用 | 军事、政府等高安全级别应用 |

**详细说明**：
- **DAC**：通过GRANT和REVOKE语句来授予和收回权限。用户对不同的数据对象有不同的存取权限，不同的用户对同一对象也有不同的权限。用户还可以将其拥有的权限转授给其他用户。
- **MAC**：对数据对象和用户都进行安全级别标记。数据对象有密级（如公开、秘密、机密、绝密），用户有许可证级别。只有用户的许可证级别 ≥ 数据对象的密级时，用户才能读取该数据对象。

### 9.2.3 SQL 的授权与收回

```sql
-- 授权（GRANT）
GRANT SELECT ON Student TO User1;  -- 授予查询权限
GRANT ALL PRIVILEGES ON Student TO User2;  -- 授予全部权限
GRANT SELECT ON SC TO User3 WITH GRANT OPTION;  -- 授权并允许转授

-- 收回（REVOKE）
REVOKE SELECT ON Student FROM User1;  -- 收回查询权限
REVOKE ALL PRIVILEGES ON Student FROM User2;  -- 收回全部权限
```

### 【例题1】（2020年下半年）

**题目**：在SQL中，用于授予权限的语句是（）。
A. GRANT
B. REVOKE
C. CREATE
D. DROP

**答案**：A

**解析**：GRANT用于授予权限，REVOKE用于收回权限，CREATE用于创建对象，DROP用于删除对象。

### 【例题2】（2022年上半年）

**题目**：以下关于DAC和MAC的说法，正确的是（）。
A. DAC比MAC更安全
B. MAC中用户可以自主地将权限授予其他用户
C. DAC中用户可以自主地将权限授予其他用户
D. DAC和MAC没有区别

**答案**：C

**解析**：DAC（自主存取控制）允许用户自主地将权限授予和收回，灵活但安全性较低。MAC（强制存取控制）由系统强制实施，安全性高但灵活性差。

---

## 9.3 视图机制

### 9.3.1 概述

**教材定义**：为不同的用户定义不同的视图，把要保密的数据对无权用户隐藏起来。

**通俗解释**：视图可以用来做"数据脱敏"——让用户只看到他们应该看到的数据，隐藏敏感数据。

**示例**：
```sql
-- 只让学生看到自己的信息
CREATE VIEW MyInfo AS
SELECT Sno, Sname, Sage
FROM Student
WHERE Sno = CURRENT_USER;

-- 只让教务处看到学生成绩
CREATE VIEW ScoreView AS
SELECT Sname, Cname, Grade
FROM Student, SC, Course
WHERE Student.Sno = SC.Sno AND SC.Cno = Course.Cno;
```

## 9.4 审计

### 9.4.1 概述

**教材定义**：审计功能把用户对数据库的所有操作自动记录下来放入审计日志中。DBA可以利用审计日志重现导致数据库现有状况的一系列事件，找出非法存取数据的人、时间和内容。

**通俗解释**：审计就是"录像"——记录所有数据库操作，出了问题可以回看是谁干了什么。

### 9.4.2 审计的类型

| 类型 | 说明 |
|------|------|
| 用户级审计 | 针对用户自己创建的表或视图进行审计 |
| 系统级审计 | DBA设置，监测grant、revoke等操作 |

**审计的优点**：事后追踪，能找出非法操作的用户和操作内容。
**审计的缺点**：增加系统开销，审计记录占用存储空间。

## 9.5 数据加密

### 9.5.1 概述

**教材定义**：数据加密是防止数据库中数据在存储和传输中失密的有效手段。

**通俗解释**：加密就是把数据"打码"——即使数据被偷了，没有密钥也看不懂。

### 9.5.2 加密方法

| 方法 | 说明 |
|------|------|
| 替换方法 | 使用密钥将明文中的每一个字符转换为密文中的一个字符 |
| 置换方法 | 将明文中的字符按不同的顺序重新排列 |
| 混合方法 | 将替换和置换结合使用 |

### 9.5.3 加密层次

| 层次 | 说明 |
|------|------|
| OS层加密 | 在操作系统层面对文件进行加密 |
| DBMS内核层加密 | 在数据物理存取之前完成加解密，对用户透明 |
| DBMS外层加密 | 在应用程序中完成加解密 |

### 9.5.4 安全机制总结

| 安全级别 | 措施 | 作用 |
|---------|------|------|
| 最外层 | 用户标识与鉴别 | 验证用户身份 |
| 第二层 | 存取控制（DAC/MAC） | 控制用户权限 |
| 第三层 | 视图机制 | 隐藏敏感数据 |
| 第四层 | 审计 | 记录操作日志 |
| 第五层 | 数据加密 | 保护数据本身 |

### 【例题1】（2019年下半年）

**题目**：数据库安全性控制从外到内的层次依次是（）。
A. 用户标识与鉴别、存取控制、视图机制、审计、数据加密
B. 存取控制、用户标识与鉴别、审计、视图机制、数据加密
C. 数据加密、审计、视图机制、存取控制、用户标识与鉴别
D. 视图机制、存取控制、用户标识与鉴别、审计、数据加密

**答案**：A

**解析**：数据库安全性控制从外到内依次是：用户标识与鉴别（最外层）、存取控制、视图机制、审计、数据加密（最内层）。

**题目**：审计功能的主要作用是（）。
A. 防止非法用户登录
B. 控制用户权限

**解析**：审计功能将用户对数据库的所有操作自动记录到审计日志中，便于事后追踪和审计。A是用户标识与鉴别的作用，B是存取控制的作用，D是数据加密的作用。

## C.1 范式判断公式

### 判断流程

```
开始
  ↓
属性是否不可再分？ ─否→ 不满足1NF
  ↓是
满足1NF
  ↓
是否存在非主属性对候选码的部分依赖？ ─是→ 满足1NF，不满足2NF
  ↓否
满足2NF
  ↓
是否存在非主属性对候选码的传递依赖？ ─是→ 满足2NF，不满足3NF
  ↓否
满足3NF
  ↓
每个决定因素是否都包含候选码？ ─否→ 满足3NF，不满足BCNF
  ↓是
满足BCNF
```

### 关键判断条件

| 范式 | 满足条件 | 消除的问题 |
|------|---------|-----------|
| 1NF | 每个属性不可再分 | 重复组、复合属性 |
| 2NF | 1NF + 不存在非主属性的部分依赖 | 非主属性部分依赖 |
| 3NF | 2NF + 不存在非主属性的传递依赖 | 非主属性传递依赖 |
| BCNF | 3NF + 每个决定因素都含候选码 | 主属性的部分/传递依赖 |

## C.2 码的求解公式

### 步骤

1. 将属性分为三类：
   - **L类**：只出现在函数依赖左边的属性（一定在候选码中）
   - **R类**：只出现在函数依赖右边的属性（一定不在候选码中）
   - **N类**：不在任何函数依赖中出现的属性（一定在候选码中）
   - **LR类**：既出现在左边又出现在右边的属性（可能在候选码中）

2. 用L∪N类属性闭包，若能推导出所有属性，则L∪N就是唯一候选码
3. 若不能，依次加入LR类属性，求闭包，找到最小的候选码

### 闭包计算

给定属性集X和函数依赖集F，X的闭包 X⁺ 是由X出发，利用F中的函数依赖所能推导出的所有属性的集合。

**算法**：
```
X⁺ = X
repeat
    for each 函数依赖 Y→Z in F
        if Y ⊆ X⁺ then
            X⁺ = X⁺ ∪ Z
until X⁺ 不再变化
```

## C.3 关系代数常用公式

| 运算 | 符号 | 性质 |
|------|------|------|
| 选择 | σ_F(R) | 行操作；σ_{F1 AND F2}(R) = σ_{F1}(σ_{F2}(R)) |
| 投影 | π_A(R) | 列操作；π_{A}(π_{A,B}(R)) = π_{A}(R) |
| 笛卡尔积 | R×S | 属性数=相加，元组数=相乘 |
| 自然连接 | R⋈S | 等值连接+去重复公共列 |
| 选择与投影交换 | π_A(σ_F(R)) = σ_F(π_A(R)) | 当F只涉及A中的属性时 |

## C.4 无损连接判定公式

R分解为R1和R2的无损连接判定：

**充要条件**：(R1 ∩ R2) → (R1 − R2) 或 (R1 ∩ R2) → (R2 − R1)

即：两个子模式的公共属性必须能函数决定其中至少一个子模式的全部属性。

## C.5 封锁协议对比公式

| 协议 | 写操作 | 读操作 | 防丢失更新 | 防读脏数据 | 防不可重复读 |
|------|--------|--------|-----------|-----------|-------------|
| 一级 | 加X锁，事务结束释放 | 不加锁 | ✓ | ✗ | ✗ |
| 二级 | 加X锁，事务结束释放 | 加S锁，读完释放 | ✓ | ✓ | ✗ |
| 三级 | 加X锁，事务结束释放 | 加S锁，事务结束释放 | ✓ | ✓ | ✓ |

## C.6 故障恢复策略

| 故障类型 | 恢复策略 |
|---------|---------|
| 事务故障 | UNDO（撤销该事务所有修改） |
| 系统故障 | UNDO未完成事务 + REDO已提交事务 |
| 介质故障 | 装入最近备份 + REDO已提交事务 |

```
用户标识与鉴别（最外层）
    ↓
存取控制（DAC/MAC）
    ↓
视图机制
    ↓
审计
    ↓

```
